# 规模诊断：镇子扩到 N 会怎样（含一次被自己抓到的过度解读）

分支 `scale-diagnostic`。#15 结案指出"共识放逐是扩容才谈得上的涌现"，本轮用 `Sim.spawn_count`（克隆扩容，确定性）
+ 新增 `Harness --agents N` + 合并到 master 的 shadow 探针，扫 N=12/24/48/96，看①不变量+确定性②性能③放逐是否随 N 变可测。

## 诚实先行：一次过度解读及其修正
初跑（3 seed）我一度报"N=48 放逐 PASS、机制得证"。**错了**——那条 `scale_48.jsonl` 有 **25% 逐字节重复记录**：
第一轮 sweep 的 N=48 godot 被工具 120s 超时杀 bash 后，**子进程僵尸存活继续写同一文件**，与第二轮并发写者交叠 → 记录翻倍
→ 把某 outcast 的窗口内弱关系接触从真值 4 吹成 8（够到可评门槛）→ 假 PASS。**加种子firm-up（3→8 seed）时对不上，才揪出来。**
教训：①`--shadow-dump` 有并发写者污染风险（僵尸 godot），跑之间要杀僵尸、别用同名 dump 路径；②分析器已加【按 (seed,tick,actor,target,action) 去重】防护；③单 seed 的强结论不可信，firm-up 是必须的。

## 修正后的真实结果
**① 不变量 + 确定性**：N=48 全绿（#01 无饿穿 **8/8**、det 2/2 逐字节一致）。**N=96 破 #01**：无饿穿 **4/6**（seed 3、5 饿穿）。
根因=**资源密度不随 N 扩**：全镇约 8 个带 advertises 的满足物 + 地图饭点，喂 12 人够、喂 96 人不够 → 高 N 下抢不到满足者饿穿。N=48 是当前安全上限。

**② 性能**（3 seed × 60 天 wall）：N=12→18s、24→39s、48→112s、96→461s。到 N=48 亚二次（R1/R2 LOD 有效）；N=96 转超线性（~4×/2× agents），叠加饿穿抖动。

**③ 放逐是否随 N 变可测**（clean 单写者 trace，窗口内弱关系 greet/invite 最大接触数，门槛=8）：

| N | seeds | outcast 状态占比 | 单 outcast 最大窗口内接触 | 达到可评(≥8)? |
|---|---|---|---|---|
| 12 | 3 | 5.1% | 2 | 否 |
| 24 | 3 | 4.1% | 4 | 否 |
| **48** | **8** | 3.3% | **6** | **否（8/8 INCONCLUSIVE）** |
| 96 | 3 | 2.9% | 4 | 否 |

**接触数随 N 增长(2→4→6)但到 N=96 都没稳定越过可评门槛。** 且 outcast 状态占比随 N【下降】(5.1%→2.9%)——固定"≥2/3 的人讨厌你"门槛在更大镇里更难达成。所以：**共识放逐在 N≤96、当前镇结构下仍不可靠地可测**；#15 结案的"扩容才可测"是【方向对但门槛未到】——要么需要更大 N/更聚簇的镇结构（派系/公共场所加剧对立），要么 #15v2 的 outcast 门槛该随规模自适应。这条留给后续。

## 落地与结论
- `Harness --agents N`（Sim.spawn_count；默认 0 逐字节不变）+ 分析器去重防护。
- **安全扩容上限当前 = N≈48**（全绿+确定+亚二次）。**要冲更高 N，先修资源密度随人口缩放**（高 spawn_count 时按比例多放满足物/饭点/床），这是解锁"数百 NPC 真活着"的必要前置。
- 放逐可测性：方向对（接触随 N 涨），门槛未到（N≤96 仍 INCONCLUSIVE）。不改机制；#15v2 保持诊断。

_3-8 seed 探针，非大样本；N=96 仅 3-6 seed。结论为方向性。master 代码未动（本分支 + 已合并的 shadow 探针）。_
