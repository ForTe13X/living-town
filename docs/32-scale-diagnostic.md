# 规模诊断：镇子扩到 N 会怎样（含一次被自己抓到的过度解读）

分支 `scale-diagnostic`。#15 结案指出"共识放逐是扩容才谈得上的涌现"，本轮用 `Sim.spawn_count`（克隆扩容，确定性）
+ 新增 `Harness --agents N` + 合并到 master 的 shadow 探针，扫 N=12/24/48/96，看①不变量+确定性②性能③放逐是否随 N 变可测。

## 诚实先行：一次过度解读及其修正
初跑（3 seed）我一度报"N=48 放逐 PASS、机制得证"。**错了**——那条 `scale_48.jsonl` 有 **25% 逐字节重复记录**：
第一轮 sweep 的 N=48 godot 被工具 120s 超时杀 bash 后，**子进程僵尸存活继续写同一文件**，与第二轮并发写者交叠 → 记录翻倍
→ 把某 outcast 的窗口内弱关系接触从真值 4 吹成 8（够到可评门槛）→ 假 PASS。**加种子firm-up（3→8 seed）时对不上，才揪出来。**
教训：①`--shadow-dump` 有并发写者污染风险（僵尸 godot），跑之间要杀僵尸、别用同名 dump 路径；②分析器已加【按 (seed,tick,actor,target,action) 去重】防护；③单 seed 的强结论不可信，firm-up 是必须的。

## 修正后的真实结果
**① 不变量 + 确定性**：N=48 全绿（#01 无饿穿 **8/8**、det 2/2 逐字节一致）。**N=96 破 #01**：无饿穿 **4/6**（seed 3、5 饿穿）。
根因=**资源密度不随 N 扩**：全镇约 8 个带 advertises 的满足物 + 地图饭点，喂 12 人够、喂 96 人不够 → 高 N 下抢不到满足者饿穿。N=48 是当前安全上限。

**② 性能**（3 seed × 60 天 wall）：N=12→18s、24→39s、48→112s、96→461s。到 N=48 亚二次（R1/R2 LOD 有效）；N=96 转超线性（~4×/2× agents），叠加饿穿抖动。

**③ 放逐是否随 N 变可测**（clean 单写者 trace，窗口内弱关系 greet/invite 最大接触数，门槛=8）：

| N | seeds | outcast 状态占比 | 单 outcast 最大窗口内接触 | 达到可评(≥8)? |
|---|---|---|---|---|
| 12 | 3 | 5.1% | 2 | 否 |
| 24 | 3 | 4.1% | 4 | 否 |
| **48** | **8** | 3.3% | **6** | **否（8/8 INCONCLUSIVE）** |
| 96 | 3 | 2.9% | 4 | 否 |

**接触数随 N 增长(2→4→6)但到 N=96 都没稳定越过可评门槛。** 且 outcast 状态占比随 N【下降】(5.1%→2.9%)——固定"≥2/3 的人讨厌你"门槛在更大镇里更难达成。所以：**共识放逐在 N≤96、当前镇结构下仍不可靠地可测**；#15 结案的"扩容才可测"是【方向对但门槛未到】——要么需要更大 N/更聚簇的镇结构（派系/公共场所加剧对立），要么 #15v2 的 outcast 门槛该随规模自适应。这条留给后续。

## 落地与结论
- `Harness --agents N`（Sim.spawn_count；默认 0 逐字节不变）+ 分析器去重防护。
- **安全扩容上限当前 = N≈48**（全绿+确定+亚二次）。**要冲更高 N，先修资源密度随人口缩放**（高 spawn_count 时按比例多放满足物/饭点/床），这是解锁"数百 NPC 真活着"的必要前置。
- 放逐可测性：方向对（接触随 N 涨），门槛未到（N≤96 仍 INCONCLUSIVE）。不改机制；#15v2 保持诊断。

_3-8 seed 探针，非大样本；N=96 仅 3-6 seed。结论为方向性。master 代码未动（本分支 + 已合并的 shadow 探针）。_

## Codex desktop 评审校正（全仓核验，只读；重要，改了本诊断的可信度边界）
把本轮送 Codex 对抗评审，它抓出几处让结论必须收窄的方法学问题——都属实，逐条校正：
1. **资源密度是【未证实】的推断，不是确诊**：我从"1 个饭点喂 96 人"推出"资源密度导致 #01"，但【没确认到底哪个 need 触底】（Harness 的 #01 只报"触底 need·tick=9"，不报 need 名）。可能是 hunger 抢不到饭点，也可能是寻路/travel-time/对象占用排队/energy(床)等。**须先加 need-分解遥测确认哪个 need 饿穿，再谈修法**。当前"扩满足物密度"只是候选修法，未验证根因。
2. **规模测试跑的是【全量 sim、没开 LOD】**——而这【不是】游戏真跑数百 NPC 的方式：R1/R2 的 LOD（远端聚合/降频）才是。所以 N=96 全量 sim 的性能(461s)与 #01 破，可能【高估成本、也不代表实际出货行为】。规模结论应在 `LodAblation.gd` 的 LOD 配置下重测才作数。
3. **克隆器改了社交动态**（Sim.gd:497-500）：spawn_count 克隆用 persona 轮转 + 按区心 spawn，改变了社交图结构（相同 persona 扎堆、天生立场雷同、空间聚集）。**"放逐随 N 变可测"这个结论来自【克隆镇】，不可靠**——克隆镇 vs 真实多样镇的社交/放逐动态未证等价。
4. **shadow-dump 的去重远不够**：exile_v2 的 (seed,tick,actor,target,action) key 缺 run-id/N/config/commit，相同 key 即使 accepted/margin 不同也静默 keep-first，可能把不同配置拼成"Frankenstein trace"、且救不了 truncate/丢行/半个 seed。**真修法=--shadow-dump 每 run 独立文件 + 头部写 manifest(commit/N/config)**，不是靠事后去重。
5. **门槛口径**：固定"≥2/3 全镇负向"(global consensus)不适合几十人尺度、更不适合更大镇（越大越难让 2/3 讨厌你）。要玩家可见的放逐，应转向【局部】放逐口径 `#15-local`（预注册），而非全局共识。

**⇒ 校正后的诚实结论**：本规模线**只到"根因方向诊断"**，不构成"放逐随规模可测/机制得证"的结论（克隆伪影 + 无 LOD + 资源根因未证 + global 门槛不适配，四重削弱）。**产品方向岔路**（Codex 也点明）：①若目标是【当前几十人的 town】——这条线到此为止，够了；②若目标是【数百 NPC】——继续，但要**重定义为"规模与空间架构工程"（LOD 配置下重测 + need-分解确认 + 克隆等价性验证 + dump 鲁棒化 + #15-local 口径），不再以"证明 exile"为目标**。这是需要产品拍板的方向选择，非纯工程。

## 规模工程（用户选②：数百 NPC）· 两个根因诊断已证
按 Codex 前两条纠正做了验证，结果【推翻了我之前的判断，也大大简化了任务】：

**A. N=96 #01 根因【已证实】= hunger 食物点争抢**（`find_starve.gd -- 3 96 60`）：seed3 只有 **1/96** agent 饿穿，全是 **hunger**，且饿穿时正 **在食物点(吃饭@40,20)吃饭** hunger=0.44。即：全镇 1 个饭点喂不动 96 人 → 挤在饭点还是把 hunger 掉到 0.5 下。资源密度假说【成立】。

**B. 关键：出货的 LOD 路径在 N=96 【本就没事】**（`LodAblation.gd --agents 96 --gate agg`）：三档(off/保守/激进)**饿穿累计全 0**、**硬不变量全绿**、激进档 cand_calls 省 **85%**(30672→4488)、**激进门 PASS ✅、LOD ABLATION PASS ✅**。原因：激进 LOD 把远端 agent 变【聚合背景群演】——不全量跑 need/寻路/对象占用 → 不挤饭点 → 无争抢 → 不饿穿。**所以我之前"N=96 破 #01"是【全量 sim 专属伪影】，不是出货路径的 bug**（Codex 第②条完全正确）。

**⇒ 重大简化**："数百 NPC 真活着"的能力【其实已经具备】——出货的激进 LOD 路径 N=96 零饿穿、硬不变量绿、成本降 85%（这是 R1/R2 LOD 工作 task#21 的既有成果，LodAblation 一直在 PASS）。全量 sim 的食物争抢只是【近镜头/无 LOD 情形】的问题，不阻塞出货。**真正的剩余工作**：①把 LOD 路径推到 N=200/500 找真天花板（进行中）；②可选：修全量 sim 食物密度(高 spawn_count 时按比例加饭点)，让近镜头一簇人也不饿穿；③远端"群演"够不够"活"(不然远处像死城)。**exile 可测性不再是目标**（用户已把方向定为规模工程）。
