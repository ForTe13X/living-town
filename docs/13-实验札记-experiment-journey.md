## 2026-07-05 · GPU 之谜坐实(CPU-only) + 异步启动探测(真机验证不再黑屏)

💡🛠 两件事收口。**① GPU vs CPU 坐实**：三路证据都指 **NobodyWho v9.4.0 的 arm64-android 是 CPU-only**——(a) 反查我们自己那个 `.so`：`grep -a` 里 vulkan/ggml_vulkan/opencl/adreno **全 0**，而 Windows dll 是 vulkan×16/ggml_vulkan×13（且真跑到 AMD Vulkan）；(b) NobodyWho `Cargo.toml` 的 `vulkan` feature 明确 `not(target_os="android")`，安卓只给 `android-static-stdcxx`；(c) 官方文档白纸黑字"Mobile platforms currently run CPU-only"。`use_gpu_if_available` 在安卓是**空操作**。**关键洞察**：我们量到的 ~40s/发 比 3B CPU decode 还慢 → 是 **prefill 主导**（长 system prompt + grammar），不是纯 decode。所以端上要能用：**0.5B(稳)/1.5B(勉强)** + 砍 prompt + **Q4_0**(ARM 有快 repack GEMV，比 Q4_K_M 快)。GPU 想要得换 OpenCL-Adreno 自编 llama.cpp + 别的绑定，且只快 prefill 不快 decode。

⚠️🛠 **② 异步启动探测**：真机上"启动进 slm 黑屏 85s"的根是 `Main._ready` 在建 WorldView/HUD **之前** `await probe_capability`（1.9GB 模型 load+2 暖发）。修法：**先建视图/HUD、镇子立刻跑 logic 地板，探测改后台异步**——`probe_capability(be,…)` 收显式目标后端，**够快才切 be**（原来是"太慢降 logic"，现在反过来"够快才升 slm"），boot 与运行期 toggle 共用 `_probe_and_activate`。**真机验证**：uninstall→install debug→授权→toggle slm→重启进 slm，8s 就出镇子（旧版黑 85s），状态栏 logic、日志印"模型加载+探测中…镇子先跑 logic 地板"，送礼社交照流——**黑屏没了、镇子探测中也不冻**。顺带修了 toggle 进 slm 不重探测→静默 100% 超时冻镇（现在 toggle 也走 `_probe_and_activate`）。**桌面红线守住**：默认 logic digest 3858030099 逐字节不变、S0 全绿（异步探测只在窗口 slm/llm 路，不碰 CI）。

## 2026-07-12 · 决策≠生成：把语音从决策里拆出来，端上 slm 一下快 2.4-2.8×

💡🔄 姊妹 lab `Dev/edge-npu-8elite` 一句话点醒：**决策 = 从 N 候选选 1 = 一次 prefill + argmax = ≈0 decode**；开放生成（语音）才是 decode 重活。小镇一直把两者揉进一次 slm 调用（JSON `{pick,speech,...}` ~80 decode token）——9.7s 的元凶就是这 80 token 的 decode。拆开：模型只回一个**编号字符**（decode≈1 token），**台词全走本会话早先建的冻结·70B 语音库**。**严格对拍（同 1.5B/同探针/同机）**：桌面 Vulkan **365→150ms(2.4×)**、桌面 CPU **985→349ms(2.8×)**——省下的正是塌掉的 decode（CPU decode 慢→省更多→CPU 收益更大，预示手机受益最大），且 slm 由此**激活**（<8s 门）。**手机 matched-cold 实测（CPU 起 35°C 两跑，后补）：OLD 14768ms → NEW 12667ms，只 1.17×**——远不及桌面 2.4-2.8×。⚠🔄 **反直觉真相：手机是 prefill-bound、桌面是 decode-bound**——手机 CPU 上 12 居民候选 prompt 的 prefill(~327tok) 就吃 ~12.7s，80-token decode 只 ~2.1s；decouple 砍的正是 decode，prefill 没动。**我先前"外推冷机 ~3.4s 激活"是错的**（假设手机 decode 占比≈桌面，实则瓶颈翻转）。**教训**：①跨设备别按比例外推延迟结构，瓶颈(prefill vs decode)会翻转，只有 matched 实测算数；②热态要读 CPU 分区(cpu-*/cpuss-*~32°C)不是 PMIC 分区(pmih010x 61-79°C，我误读成"热")；③工具坑连环：adb 端口漂移(mDNS 重发现)、Godot print 不进 logcat(改写 Documents 文件 pull)、am start 不唤屏→tap 打空(keyevent 224 唤屏)、探针自身即热源(冷启≈暖启，长时段持续推理才是深度节流)。**结论转向**：端上 slm 真跑起来，方向不是"更巧的 decouple"，是"决策 prefill 搬 NPU(edge-npu 路，NPU prefill-强正补 CPU 短板)"或"决策留 logic + 语音走冻结 70B(本会话已成)"。详见 [docs/21](21-decision-voice-decouple.md)。**教训**：①一个看似"多模态输出"的 LLM 调用，把 fast-path(分类)和 slow-path(生成)揉一起是延迟结构的原罪，拆开是数量级的事；②rigor 靠术语等级(resident-warm/cold/thermal-hot)+受控 A/B+诚实标"不可比/外推"，别拿热态混淆的数当 A/B。详见 [docs/21](21-decision-voice-decouple.md)、[[reference-edge-npu-8elite]]。红线：decouple 是 slm/llm-only，S0 digest 未动。

## 2026-07-11 · 真机标定（1.5B live 探测）
**真机标定（1.5B live 探测）**：推 qwen2.5-1.5B-Q4_K_M 到设备、重启进 slm live 探测——**p50=9680ms（~9.7s/发）→ demoted_logic**（8s 线就差一点点）。3B 是 ~40s，1.5B 是 9.7s。**关键**：9.7s 里绝大半是 **12 居民长 prompt 的 prefill**（research 预言中）——不是 decode。所以过线的最省路是**砍 prompt/缓存静态前缀**，或换 **0.5B**（~2-3x 快 → 3-5s → 稳过），或把 demote 阈值放宽到接受 ~10s/发（能跑但 LLM 节奏慢）。异步探测期间镇子全程 logic 地板活着、24s 截图还在探、30s 才出 demoted——**首帧不黑、探测不冻，真机实锤**。

## 2026-07-05 · adb 真机环打通 → 一口气挖出端上 SLM 全貌（4 个真机专有 bug）

🍀🛠 用户搭了 `Dev/godot-android-loop`（无线 adb：export→install→launch→logcat 一条命令 + `screencap` 给 agent 看），**桌面再也验不到的端侧终于可驱动**。一轮 deploy→screencap→Read→logcat 就把"手机上 SLM 到底行不行"从完全黑箱挖成全貌：
- ✅ **美术真机加载正常**（地形/精灵/家具全在）——之前那个"导出剥裸资源"的修复真机坐实。✅ **12 独立居民**真机各异渲染。✅ 模型文件解析到（`✓ 就位 /Documents/model.gguf`，1.93GB）。
- ⚠️ **但 SLM 100% 超时**：perf overlay `发起 68 · 成功 0 · 超时 66`，且**内存仅 176MB**——2GB 的模型根本没 load 进内存。🔑 根因 `appops`：**`MANAGE_EXTERNAL_STORAGE: default; rejected`**——权限被拒 → 公共 Documents 的 gguf 能 `stat` 却不能 `mmap` 读 → 模型永远载不上 → 每发必超时。`adb shell appops set … allow` 后重启，`dumpsys meminfo` **TOTAL PSS 2.43GB（~1.9GB clean=模型 mmap）**——模型真载上了。
- 🚧 **但启动进 slm 会黑屏 ~85s**：`Main._ready` 在建 WorldView/HUD **之前** `await probe_capability`（载模型+2 发暖推理），1.9GB 模型在真机上这一步就是 85s 黑屏。**修法：先建视图再异步探测。**
- 🚧 **运行期 toggle 进 slm 不重探测**（我当初为省事跳过的）→ 无自适应截止线、无降级 → 模型慢/坏时**静默 100% 超时、镇子冻住**。而**启动进 slm 会探测** → 正确降级：真机 probe 测出 3B **p50≫8s**（~40s/发，几乎肯定 arm64 无 Adreno GPU、纯 CPU）→ 自动降 logic，镇子照活。
- 🐛 小 bug 两枚：`list_models` 的 `DirAccess` 枚举公共 scoped-storage 目录失败 → 设置里显示"无 gguf"（但 `file_exists` 解析成功，故照跑）；设置面板后端显示是启动时的旧值（stale）。

**结晶**：① 端侧 gguf 别放公共 Documents（要 all-files 权限，多数用户会拒）——放 app 私有 `user://`(Android/data/<pkg>/files) 免权限，或运行期申请权限。② **3B 在这机器上太慢**（CPU），端上要 **1.5B/0.5B** 或坐实 GPU 加速。③ 别用 `await probe` 挡首帧。④ **真机专有 bug 桌面一个都测不到**——这套 adb 环是端侧 vibe-coding 的命门。见 [[reference-godot-android-loop]]。

## 2026-07-05 · 端上 SLM 真跑通了（118/120 落地）+ 一个删掉才对的 sampler

🍀🛠 用户报"slm 没声音"，一查截图——在 **llm** 后端（HTTP→127.0.0.1 LM Studio，手机上根本没有），发起 3006 全废。换个思路，本机把 slm 路径整条验一遍：NobodyWho 的 Windows dll + 真 qwen2.5-1.5b gguf，跑 `--backend slm`。**AMD Vulkan GPU 直接吃下**（arch qwen2、gpu_layers=29），算力探针 tier=fast p50~365ms，40s 内 **发起 120 · 成功 118 · 超时 0 · 无效 0**——**端上 LLM 声音是真的能跑，98% 落地**。arm64 安卓库同一套 API，手机（Adreno）只会更快。

⚠️ 顺手逮到个坑：决策路径在 `start_worker` 后调 `set_sampler_preset_constrain_with_json_schema`，但 NobodyWho worker 是**异步起**的，ask 前 worker 常没就绪 → 配置被丢，90s 刷了 **262 条** "Worker not started, dropping sampler config"。而且这正撞上代码自己早先的发现（chat 路径注释：json_schema 受限解码在长多候选 prompt 下**卡死只吐"{"**）。删掉这个"其实一直在被丢弃"的调用：警告 262→0，决策照样靠 `parse_decision` 抽 {…} 98% 落地。**教训：一个 no-op 的调用不是无害的——它既刷日志又埋着"偶尔真生效就卡死"的雷。**

## 2026-07-05 · 编辑器里好好的，导出就没美术：裸资源被 PCK 剥了

**🔄 桌面/编辑器全绿、录了无数遍 demo 的美术，一装进 APK 全成程序化兜底——因为 `res://` 在编辑器和导出物里指的根本不是一个东西。**
`Art.gd` 一直用 `Image.load("res://…png")` + `FileAccess.file_exists()` 读**裸 png**。编辑器里 `res://` 就是磁盘工程目录、裸文件在，所以一路绿到底、demo 录了一堆。可一导出 APK，镇子全没纹理：**导入过的 png 被转成 `.ctex`、裸 png 在导出 PCK 里被剥掉，而 `FileAccess`/`Image.load` 这类低层文件操作【不跟 `.import` 重映射走】**——于是 `file_exists` 全 false、`tex()` 全 null、退程序化。字体 `cjk.ttf` 同病（CJK 会变豆腐）。修法：改走 `ResourceLoader.exists()+load()`（它跟重映射取到**打包进去的** `.ctex`/FontFile，编辑器/导出都对），裸文件解码只留作 `pro/` 覆盖的兜底。用带窗口真 framebuffer 渲一帧坐实：地形/精灵/家具/中文字全出。
**为什么重要**：这是"编辑器 ≠ 导出"里最阴的一类——`res://` 两处语义不同（磁盘工程 vs PCK），凡"运行时按路径读资源字节"的代码（图、字体、任意 data 文件）都可能编辑器绿、真机空。**教训**：① 任何 `res://` 运行时加载，验证必须在【导出产物】上做，不能只在编辑器/桌面跑；② 能 `load()` 就别 `FileAccess`+手动解码——`load()` 天生跟导入系统走。附带补了个 `--shot <abs.png>` 出图 hook（skill 头号推荐的验证钩子）——**正因为一直缺它、只在桌面肉眼看，这个 bug 才在导出物上藏了这么久**。

## 2026-07-05 · 手机可用性三连：新功能悄悄拆了老保险 · MTP 藏路径 · daemon 吞管道

**🔄 加一个"运行期切后端"的开关，把一条本来好好的安全阀（太慢自动降 logic）悄悄拆了——对抗式评审替我逮住的。**
手机上没 CLic，得能在应用内点按切 logic↔slm。我把 `backend`(生效档) 和 `backend_requested`(用户意图) 拆开，`decide()` 在无在飞请求的安全点才把意图应用上去——想得挺周全，12-seed digest 也逐字节没变，就交了。结果 4 维对抗评审（15 findings，13 refuted）留下 1 条真的：算力探测测到 `>8s` 会把 `backend` 降成 logic 保护慢机，但**没同步 `backend_requested`**；于是第一帧 `decide()` 一看"生效≠意图"，立刻把慢后端又切回来——**降级被无声撤销，慢机每个决策空烧 12s 才超时回落**。这是"加新功能时，新逻辑和一条既有的、你没在想的旧逻辑在同一个字段上打架"的典型。修法一行：降级时也同步 `backend_requested`。**教训**：一个改动只要碰了某个别处也在写的状态，就得把那个状态的**所有**写者都过一遍——单测/digest 不变 ≠ 语义不变，这种"两条正确逻辑各自没错、凑一起错"的 bug 只有把交互点摆出来才看得见（所以值得专门放一轮对抗 review）。

**💡 端上模型放哪，被现代 Android 的"作用域存储"重新定义了：`user://` 你自己够不着。**
本来 gguf 走 `user://model.gguf`（app 私有外部 files 目录），文档也写好 `adb push` 到 `Android/data/<pkg>/files/`。用户一句"我 MTP 只能看到 `此电脑\手机\Documents`"点醒我：**Android 11+ 把 `Android/data/<pkg>` 对 MTP/资源管理器藏了**，用户根本拷不进去。于是把模型解析改成先找公共可拖放位置（`Documents/…`、`Download/`）再退 `user://`，并加 `MANAGE_EXTERNAL_STORAGE`（"所有文件访问"，得在设置里手动开一次——此权限不能运行时弹窗）。**为什么重要**："能把文件送到 app 能读的地方"在现代 Android 上是个真问题，不是随手 `adb push` 就完事；免 adb 的路径（公共目录 + 一次性授权）才是给非开发者用户的默认。顺带给无控制台的手机加了 boot log 打印"模型✓就位/未找到+路径"——端上调试全靠这一行。

**⚠️ 后台任务"永远在跑"，其实 APK 两分钟前就好了：gradle daemon 继承了任务的输出管道死不放。**
release APK 的后台构建挂了 1.5 小时没动静，我差点当它卡死。一查：`godot.exe` 早没了、APK 早在启动后 ~2 分钟就 117MB 落盘、日志停在同一时刻——是 `godot --export` 起的 **gradle daemon(java) 继承了后台任务的 stdout 管道并一直攥着**，harness 看管道不闭就以为任务还在跑。修法：`game/android/build/gradle.properties` 加 `org.gradle.daemon=false`，重建 ~36 秒干净退出、无僵尸。**教训**：判断长任务死活别只看"任务状态/通知"，要看**它真正的产物**（APK 的 mtime+大小、日志最后活动时间）——被守护进程继承管道这类事会让"进程编排层"和"活儿本身"脱钩。

## 2026-07-04 · 选举「收获期」：一台戏，零新状态

**💡 加"选举"这么大一个玩法，引擎里没多一个新状态字段——因为 attitude 早就是选票、faction 早就是票仓，选举只是一个纯快照 + 一条事件。**
Wave 3a 选举本以为要建投票动作、票箱、计票器一整套。真做起来发现三套已落地系统把地基全铺好了：**S2 的 `attitude∈[-1,1]` 天然就是一张选票**（`|a|<0.15` 弃权=现成的 FACTION_SALIENT 死区）、**S3 的派系天然就是票仓**（faction 场景实测两派 3-3 完美分块投，一票不乱）。所以"选举"落地=一个 `_tally_election` 纯快照函数(只读 attitudes、定序、无 RNG)+ 一条 `election` 事件 + 全镇一条里程碑记忆。**这正是"收获期"的形状**：不是又造一个系统，而是几个已在跑的系统第一次**共同**产出一个更高层的东西（"镇级决定"）。计票是纯函数这点还顺手变成硬不变量 #37（票数守恒+结果自洽），因为纯快照天然可复现。
**为什么重要**：判断一个新玩法值不值得做、好不好做，先问一句"**它需要的状态是不是已经在系统里了，只是还没被读出来？**"——选举、阶层 gossip(财富被目击=belief)、住房(room.owner 早在)都是这种"读出已有状态"的收获型增量，比"造新状态"的增量便宜且稳得多（复用了全部已验证的确定性/不变量地基）。

## 2026-07-04 · 70B 当编剧：把 LLM 的不确定性"冻成数据"，红线一寸没让

**🍀 顶配愿景里那个最唬人的件——"70B 夜间编剧"——落地只花了一个离线工具 + 给 DataScenarioProvider 加个 `beliefs` 字段，因为该复用的确定性地基早在那儿了。**
一直担心"接大模型=破确定性"。真做起来发现关键就一句话（docs/15 §2.4 早写下的）：**LLM 的产出不进 sim，产出成【数据】**。`scriptwriter.gd` 在引擎外离线跑：读镇情 → 让 `llama-3.3-70b-instruct` 写一份开局戏剧 JSON → 校验净化 → 冻结落盘 → sim 只读这份冻结的 JSON、经既有 `DataScenarioProvider` 确定性注入。LLM 有多随机都无所谓——**一旦冻成文件，回放读同一份，digest 逐字节一致**（工具自带两跑自检坐实：3908418193 两次分毫不差）。非确定性被"隔离+冻结"这一手关在了引擎门外。
**最开心的意外**：70B 真读懂了人设。给它六个居民的性格标签，它写出——可可(内向敏感)藏着一句"我一直很孤独，希望有人理解"的秘密、老邓(爱讲古)对 old_tales 打满分还跟小薇结成 trust80 的忘年交、阿丽↔阿本围着咖啡馆扩建互相记仇还各带一条谣言。冻结跑 40 天，这台戏自己演了起来（confront 35 / apologize 33 / 谣言传 27 次），36 条不变量一条没破。
**为什么重要**：这是整个"确定性地板 + 可插拔 LLM 皮肤"论题在**最激进那一点**上的兑现——不是小模型润色一句台词，是 70B 当编剧写整台戏，而**回放、跨端对拍、33+ 不变量 CI 全程一寸没让**。教训结晶：**要把不确定的东西塞进确定的系统，别让它在回路里实时跑，让它离线产出被冻结的数据、系统只消费数据。** 净化那道墙也用上了刚学的编译护栏教训——绝不信任 LLM 原样输出，只留合法 id/话题、越界钳制。

## 2026-07-04 · 编译期数据展开会把"JSON 写对"偷偷升级成确定性红线

**⚠️ 对抗式代码 review 抓到一整类潜伏 bug：`buildings.json` 编译管线里，一个数据笔误能静默漂 digest、崩载入半改 world、甚至每 tick 崩——离数据文件十万八千里。**
阶段2 的 `_compile_buildings` 把 JSON 直接展开进 world，我当时只顾着"别走 spawn_object"（确定性对了），却没意识到：**这条管线把"数据正确性"变成了确定性红线的一部分，却一道断言都没有**。五条 review 发现全 CONFIRMED（verifier 还在本机 Godot 4.6.2 上真跑了复现）：(1) 家具 id 撞 map.json 手写 id → 转字典时静默覆盖 → 候选集合变 → digest 漂，零报错；(2) 短 `rect`/`dpos`（`[15,1]`）→ 越界读是硬 SCRIPT ERROR 会**中止函数**，而 `rooms/objs` 是 world 活引用、崩前已就地半改 → world 半成品还继续往下载；(3) `furniture: {}` 写成对象 → `as Array` invalid cast 同样中止半改；(4) advertises 漏 `duration` → 编译期不崩，却在**每 tick 的 `_object_candidates` 里裸括号访问崩**，最阴——故障点离数据源最远；(5) 非对象房间/家具项 → 类型化赋值崩。
**为什么重要**：off 门/12-seed digest 这些关卡**都用干净数据测**，抓不到"改坏数据才发作"的这类。护栏得加在**编译边界**：畸形项 `push_error + continue`（不是 `return`——活引用早退同样留半改），错类型的数组字段用 `_as_arr` 退化成空，advertises 裸括号换 `.get()`。判据是 **"这东西是世界地形还是运行事件"** 的同款直觉延伸——**"缺文件=零扰动"的 off 门只覆盖了'整份缺失'，没覆盖'存在但写错'**。合规数据下全部护栏是 no-op（digest 逐字节不变，实测 seed9–12 与加护栏前完全相同），坏数据下从"静默腐化/崩"变成"跳过+报错还能跑"。教训：**任何 data→世界 的编译步骤，都要问一句"作者手滑一个字符会怎样"，并让它响亮而安全地失败。**

## 2026-07-02 · 玩家能动性 M1：~200 行就入了社交图，因为该复用的早就都在

**💡 "玩家=一个 is_player agent"是杠杆最大的 gameplay 决定：一行 append 换来整个涌现社会对玩家生效。**
没有为玩家写任何专属社交系统——玩家进 `Sim.agents` 走同一套 SocialTransaction：NPC 的接受规则会拒绝坏名声的你(实测 affinity -90 被婉拒且你"记住被拒")、旁观者围观你的打招呼、gossip 走知识边界(对方 belief 带 source=player)、约见=物理走到场即兑现(零新机器,`_resolve_commitments` 本来就只查"人在不在区里")、爽约会被积怨→NPC 来找你对质→你可以道歉。**引擎侧一共 ~200 行(大半是 player_mediate)，其余全是复用。垂直切片时"把交互对象做成一等 agent"比"给玩家开后门 API"便宜一个数量级。**

**💵 digest 只记"发生了什么"不记"多少钱"——数值系统因此能悄悄叠上去，只通过行为后果扰动回放。**
技能加成改的是工资**金额**（Lv1 木匠 +1 币），而 `event_log` 的 pay 事件只记 actor/target/tick、**不记 amount**。所以"熟练工挣更多"这件事**不直接进 digest**——它只通过"钱多了→'穷则思工'动机变了→决策变了→事件流变了"这条间接链影响回放。这解释了为什么经济/技能这类**数值**系统能一层层叠而不需要每次重录回放基线的算术部分：**digest 的粒度是"事件结构"而非"数值大小"，是刻意设计的粗粒度——它让金额、需求值、坐标这些连续量自由演化，只把"谁对谁做了什么、成没成"这种离散事实钉死。想加一个数值系统前先问：它是改事件的『结构』还是只改『数值』？只改数值的，回放天然更宽容。**

**🚪 室内的第一价值不是"新空间"，是给一个已经在跑的机制补上它一直缺的舞台。**
建筑/室内四线设计深研收敛到一句反直觉的话：**别给 `pos` 加 z 坐标（回放杀手），因为 `digest` 根本不序列化位置——室内是"谁能看见谁 + 渲染"问题，不是确定性问题。** 于是首切片不是"盖楼"，而是给 home 切一个 enclosed 卧室 + 让 confide/leak 只在同一私密房间发生。这一刀的价值全在于：S3 秘密博弈**早就在跑**，但秘密一直在开放广场当众吐露——一个存在已久却没人注意的叙事漏洞。**扩世界时，先找"哪个已落地的机制正踩在缺失的地板上"，比"还能加什么新系统"更值：补一块地板的杠杆远大于盖一栋新楼。** 实测 betray 场景开私密门后 confide/leak 仍 >0（没饿死）、默认场景因无种子秘密而逐字节不变（外科式），叙事漏洞平掉、零回归。

**🧨 并发跑 docker + 编辑期改文件 = 假 SCRIPT ERROR，真 bug 被淹没。**
室内首切片验证时冒出一片 `Nonexistent function 'new'`，一度以为是灾难；killed 掉三个堆积的 gamecraft 容器、单容器串行重跑，才看清真相：其一是并发容器读到我正在写一半的文件（假错），其二藏着一个真编译错（`var priv_ok := world.get(...)` 的 Variant 推不出 bool）。**多容器并发共享挂载卷时，任何"改完文件立刻起下一个验证"都可能读到半成品 → 假错淹没真错。纪律：改完 → 单容器串行验 → 验证期绝不碰文件；出一片错先 `docker kill` 清场 + 单跑一次拿干净 backtrace，再判是真是假。**

**🎥 阶段性录屏不只是"给人看"，它是 headless bench 覆盖不到的那一半验证——一录就抓出一个潜伏一整季的崩溃。**

**🏮 灯会桥接了派系，翻掉一条软不变量——涌现系统的"回归"有时是社会学，不是 bug。**

**🎛 机会地形的两个调参课：要压过日常，但别灌饱社交。**

## 2026-07-01 · L7 注册中枢：加一个场景 = 写一个 JSON

**💡 "ext=null 逐字节不变"把"会不会破坏老路径"从经验问题降级成读代码就能确认的问题。**
注册中枢的所有挂点都是 `if ext != null: ...`，而默认 `ext=null`。所以新代码在默认路径上**根本不执行** → S0 digest 与主分支逐字节相同，是结构上保证的，不靠"测了没变"。**给可选扩展设一个"默认即关闭"的门，是最省心的向后兼容——审阅者不用跑全套回归就能确信老行为没动。**（同 rhythm.json 缺文件即退化、lod 默认 false 的同一招。）

**⚠️ freeze 的幂等性是这个注册表的回放命门——对抗校验在动工前就点了名。**
`goto_tick` 会反复 `start_new`。如果内建 provider 在 `start_new` 里注册（append），每次重演都翻倍数组 → 候选序/夜间机制重复 → digest 漂移。设计阶段的红队 agent 提前揪出了这点，于是定死：**注册只在注入时做一次（sim_soak 建 ext），`start_new` 只调 `freeze()`（幂等排序，重复调安全）。** 一个"实现完才会炸"的坑，被一次纯阅读挡下。

**🍀 数据驱动场景一次就跑通——`--scenario rift` 写个 JSON 就劈出两派。**
`DataScenarioProvider` 读 `data/scenarios/rift.json`（声明式设 attitudes），零 GDScript、零改 Sim 核心，就让镇子在 cafe_expand 话题上裂成两派（派系=2、endorse/rally_oust 涌现、33/33 全绿、两跑 byte 一致）。**扩展性的试金石不是"能不能配置"，而是"加一个新东西要不要碰核心"——这里答案是不碰，只加一个文件。** 小坑：设 attitudes 时必须同步 `attitude0`（天生立场锚），否则 FJ 更新会把观点拉回旧锚、rift 当晚就被抹平。

**🛠 调参表用"值=默认"落地：既 data-driven 又逐字节不变，还能一眼证明 knob 是 live 的。**
`data/utility.json` 把接受阈值/打分权重抽出来，但初值就填代码里的今日值 → `_w(key,default)` 返回值不变 → S0 逐字节相同（数据化零成本上线）。要证明它不是死代码：把 `accept_greet` 从 0 拉到 200，greet 接受当场从 ~100 掉到 0。**数据化一张调参表的最佳落地姿势：先用"值=默认"证明零回归上线，再改一个极端值证明它真的接线——两步分别回答"没弄坏"和"真有用"。** 这张表是给 gameplay/content 阶段平衡用的单一旋钮面板。

## 2026-07-01 · R2 profile-first，抓出藏在两层循环里的真 O(N²)

**🔬 profile 先行，一眼看穿 tick-loop 才是那 617 秒——check_all 只占 1 秒。**
R1 后墙钟仍随 N 超线性涨（N=960 = 617s）。没有再猜，先给 soak 加了两个墙钟计时（tick-loop vs check_all，纯诊断不入状态）。答案立刻清晰：tick-loop 104s / check_all 0.9s @N=480——瓶颈 100% 在每 tick，不在末尾不变量。**吃过 R1 的亏（faction LOD 化几乎没动指针）后，这次先量再改，省下一整轮瞎优化。**

**⚠️ 真凶是 `_social_candidates` 里两处 `for c2 in agents`——嵌在 `for o in _nearby_agents` 内层 → O(nearby×N)=O(N²)/agent。**
gossip_rep 和 endorse 都要找一个"我有坏名声、而邻居 o 印象没那么坏"的第三方 C，于是每个邻居都把全镇扫一遍。连激进 LOD 的 12 个近端 agent 也各自 O(N²) → 全镇 O(N²)。**"每个邻居再扫一次全体"是社交 sim 里最经典的隐藏平方——它不显眼，因为两层循环隔着几十行、内层还带 early-break 看着像 O(1)。**

**🛠 auto-create 的读扫描是回放地雷——但因为 digest 只认 event_log，绕过它反而零成本地保持了字节一致。**
第一反应：预算"坏名声第三方"列表来替代内层扫描。但 `_rel()` 会 **auto-create** 关系——原扫描顺带给每个被看过的 agent 建了个零关系，这副作用是确定性状态的一部分，天真预算会改它 → 破回放。关键洞察：**零名声关系(standing=0)从不满足 ≤−2 的入选条件、也不产生任何事件，而 digest 是对 event_log 折叠的**——所以只遍历"已存在的坏名声关系"(按 agent 定序保首个匹配)得到完全相同的候选、相同的事件、相同的 digest，只是少建了一堆惰性零关系。**当"字节一致"和"消除副作用"看似冲突时，先问清楚 digest 到底哈希的是什么——常常那条副作用根本不在哈希域里。** 实测 S0 12/12+det 3/3、N=960 tick-loop 617→68s(9.1×)、标度 ~×5.9→~×2.2（≈线性）。

## 2026-07-01 · R1 夜间 LOD 化（把 O(N²) 从"全镇"缩到"相机周围"）

**🛠 far agent 本就到不了结盟门——所以"只算 near"和"算全量"逐字节等价，但便宜一个数量级。**
夜间 `_form_pacts_greedy` 是 O(N²)。直觉上"只处理 near 集"会漏掉 far agent 的结盟，改变行为。但一想：激进 LOD 下 far agent 不社交 → trust/familiarity/complement 永远不涨 → **永远达不到结盟门槛**，全量版里它们也从不结盟。所以把外内两层循环都限制到 near 集，结果完全一样、成本从 O(N²) 掉到 O(cap²)。**LOD 的正确性论证常常藏在"被省掉的那部分本来也什么都不做"里——先证明等价，再放心砍。** 实测 N=240 76→32s、N=480 407→104s。

**⚠️ 派系 LOD 化的坑：全量重建会把 far agent 的派系标签变悬空 → 破硬不变量 #25。**
`_recompute_factions` 每夜 `factions.clear()` 全量重建。天真地"只聚类 near"会让 far agent 的 `faction` 标签指向一个不再存在的桶 → #25 派系一致崩。正解：far agent 直接置为 **factionless**（attitudes 冻结、行为上从不引用派系、near 也不与 far 交互）——既自洽又零行为影响。**给全局重建型的机制做 LOD，关键不是"跳过谁"，而是"跳过的那些要留在一个合法的静止态"。**

**🔬 优化完还得再测标度——faction LOD 化几乎没动指针，暴露真凶另有其人。**
LOD 化 `_recompute_factions` 后墙钟只从 34→32s（N=240）——几乎没变。说明它其实是 O(N×medoids)、medoids 少时近 O(N)，不是主 hog；真正的 O(N²) 是 pact formation（已修）+ 残余的密度驱动 per-tick 社交（拥挤地图里 `_nearby_agents` 仍 O(N)）。**逐项 LOD 化 + 每步重测标度，才能分清"看着像 O(N²) 的"和"真的是 O(N²) 的"——否则你会花力气优化一个 O(N) 项还纳闷为什么不快。**

## 2026-07-01 · LLM voice 深化：模型缺的不是词，是"为什么"

**💡 台词听着泛，往往不是模型菜，是 prompt 没给它"为什么"。**
旧决策 prompt 只喂 人设 + `时段0.42`(裸浮点) + 候选(`greet`/`discuss` 英文键)。模型不知道自己此刻饿不饿、心情如何、和候选对象是老友还是有过节、最近发生了啥——只能挤泛泛寒暄。补上 `[此刻]黄昏，肚子有点饿`、`[状态]最想满足:社交(71)`、`[近事]想找阿本聊看法被婉拒了`、候选带关系提示 `打招呼→阿本(投缘)`，模型立刻有了 grounding。**给 LLM"皮肤"提质，第一杠杆几乎总是 context 而非模型大小——先确认它知道"现在是什么处境"，再谈换更大的模型。**

**⚠️ 中英混排的候选是给中文模型的暗坑。**
物件动作从 map.json 来是中文(晒太阳/做活)，社交动作是英文键(greet/discuss)——同一行 prompt 里中英混，中文模型读着别扭、容易夹英文回。加一张 `ACTION_ZH` 把社交动作也转中文(打招呼/聊看法/约见)，喂给模型的候选整齐划一。**喂模型的文本要"单语种"：任何一处内部英文键泄进 prompt，都可能诱发它中英混输出。**

**🛠 prompt 质量用"预览 scene"离线眼检，不必每次连模型。**
`build_prompt` 依赖 Sim 全局(取候选对象名)，`--script` 跑不了。做个 `prompt_preview.tscn`(scene 模式，autoload 可用)：跑 6 sim-日攒出关系/记忆/邻近，直接 print 出几个居民的真决策 prompt。**给 prompt 工程一个"离线眼检台"——看文本比连模型快一个数量级，改一版看一版，把模型调用留到定稿。** mock 决策管线(合法率 93.9%)顺带证明改 prompt 没动状态机。

**🌙 反思做成"引擎地板 + LLM 皮肤"两层：地板确定性合成洞察，模型润色成内心独白。**
`reflect()` 骨架空了很久。填法不是直接上 LLM，而是先给引擎地板一版：`Sim._reflect_agent` 每夜读关系/冲突/派系确定性合成一句(「我和阿本之间的疙瘩还没解开」「有些事情上我和几个人想到了一块儿」)，写进记忆。因为记忆不入 event_log/digest/不变量，S0 逐字节不变(12/12+det 3/3)。模型后端再叠一层 `AIBackend.reflect`(异步、复用 L5 令牌桶)把地板洞察+近期记忆润色成更自然的一句。**反思天然适配本项目的分层哲学：引擎决定"想到了什么"(确定、可回放、无模型也在)，模型只负责"把它说成人话"——和 decide/voice 同构。** GPU 3B 实测：阿丽近期"被可可婉拒"→内心独白「哎，可可嘛，下次再聊吧，镇上事我全知道。」——把被拒的记忆 + 她"消息最灵通"的人设揉成了一句真·内心戏。

录制 30s@60fps 用了 72s 墙钟(41% real-time)——movie 模式每帧固定 1/60s，SLM 的 585ms 决策在影片里被**压缩**成不到半秒，气泡落帧又快又稳(之前担心的"异步超时掉罐头"没发生，探针 deadline 3510ms 绰绰有余)。成片 `docs/media/voice_gpu_demo.mp4` 抓到完整弧线：第1天早晨「小薇，早上好…要不要一起去散步?」(按名字打招呼+晨间语境)→ 第2天深夜全镇睡觉(**昼夜节律直接在 demo 里可见**)→ 第3天白天「有点孤单，阿丽，想找个人聊聊，你愿意陪我喝杯茶吗?」——**孤独心情锚(`想找人说说话`)透过模型变成了有名有姓的邀约**。截图 shot-voice-gpu-lonely-invite.png / shot-voice-gpu-morning.png。**录 async-LLM 游戏 demo 优先选 movie 模式而非实时抓屏：帧时间确定 + LLM 延迟被压缩，是"作弊般"的顺风。**

## 2026-07-01 · 可扩展性 + 同频 + 更大群体（三线深研 → 落地）

**🔄 激进 LOD 的墙钟收益 = 0——我们一直在优化不是瓶颈的东西。**
`cand_calls` 探针显示激进 LOD 把候选枚举砍 83%，看着漂亮。但实测墙钟：激进 N=240 = 76.2s ≈ full-sim 77.8s，**一秒没省**。因为真瓶颈是**未 LOD 化的夜间 `_form_pacts_greedy`**（`for id: for oid:` 嵌套 O(N²)、每对 6 次 `_rel` 字典查，每夜为全体 N 跑），而候选枚举根本不是热点。**教训：省了一个漂亮的 metric 不等于省了墙钟——要测端到端的那个数（wall time），别被中间代理指标骗了。** 这直接推翻了"激进 LOD 就能冲上百 NPC"的假设。

**🔬 全 sim 与激进 LOD 都是 ~O(N²)：N 翻倍墙钟 ×4~5（60→120→240→480: 4.7→18→76→407s）。**
把它拆开才知道：candidate/social 是密度驱动 O(N²)（固定地图 → 越多人越挤 → `_nearby_agents` 每次 O(N)），但**夜间派系/结盟才是两配置共有的、LOD 摸不到的那块 O(N²)**。上界据此估：现状 ~N=150–200；修掉夜间 O(N²)（只处理 near 集）后可推向数百。**"能模拟"和"能便宜地模拟"隔着一个没人 LOD 的夜间循环。**

**⚠️ 昼夜节律的危险是"过度同步"——强节律把全镇钉到同一时段忙碌，反而杀死了社交。**
第一版节律用 1.8× 增幅（夜里睡眠强烈加成），同频信号极漂亮——但 #08 承诺、#26 派系当场挂掉：所有人同时睡同时吃 → 没有重叠的"空闲窗口" → invite/meet 归零。**修法=dampen-only（乘子 ≤ 1.0）**：偏好只让人"推迟"非当令的需求、绝不比基线更想做 → 物件吸引力不膨胀、不挤占社交，但抑制仍让行为按时段聚集。**给涌现系统加"节律"时，抑制比激励安全：压低离峰、别拔高在峰，否则同步性本身会碾平多样性。**

**⚠️ 短窗口"失败"是测量伪影：#08 在 20 天只有 4/12 通过——连没节律的基线都是。**
最终 coherence soak 用了 20 天，#08（承诺创建>0）报红。一查：**去掉节律、纯基线，20 天也只有 4/12**——承诺是低频涌现事件，20 天窗口根本不足以稳定断言（canonical Harness 用 60 天正是为此）。40 天：创建 28 全绿。**别把"罕见涌现事件在短窗口没发生"当 bug；断言稀有事件要用够长的窗口，否则你测的是运气不是正确性。**（节律确实把频率又降了 4/12→1/12@20d，但 60 天双双 12/12，在容差内；想补回聚会可给 invite 一条傍晚社交节律——留作后续。）

**💡 对抗式校验在动工前就拦下三个真错。**
三线研究每条都配一个"专挑刺"的校验 agent。收获：①揪出设计里一处**方向写反**的事实错误（harmony 是"非空即豁免"=过度豁免，不是"新场景被误判和睦"）；②钉出两个回放杀手——注册表的 **freeze 幂等性**（`goto_tick` 反复 `start_new` 会让内建 provider 数组翻倍→digest 漂移）与 **accept-modifier 必须逐动作等值复刻 `_faction_term`**（对不叠 fac 的动作返回 0）。**这些都是"实现完才会炸"的坑，被一次纯阅读的对抗 pass 提前挡下——设计阶段的红队比实现后的调试便宜一个数量级。**

**🛠 活跃工作集：把"每 tick 扫全部历史"降为"只扫未决的"。**
`_resolve_commitments`/`_has_active_meet` 原本每 tick `for c in commitments`——而 commitments 只增不删 → per-tick 成本 ∝ N·时间。加一个 `_active_commitments` 引用子集（创建时进、解算时出），全量列表仅留给不变量。同序处理活跃子集 → digest 逐字节不变（S0 验证），但长跑成本从"随历史膨胀"变成"∝ 未决数"。**服务"模拟时间拉长"的通法：任何每 tick 扫的列表，若只增不删，迟早成为长跑的隐形二次项。**

## 2026-07-01 · L6 调色板变体（6 张精灵撑起一城人）

**💡 视觉数量和资产数解耦——300 个不同的人，来自 6 张 CC0 精灵。**
刚把"模拟上百 NPC"做便宜了（L3 激进 LOD），渲染却还是 6 张脸 × 50 份复制，一眼假。给每个克隆按 id 派生一个确定性色相，48 个居民立刻各不相同（teal/紫/品红/蓝/红/黄绿），零新增 PNG、零版权。**scale 不只是"跑得动"，还得"看着像一城人"——两条线要一起推。** 命名 6 人零位移保正典、6 人小镇里整层休眠，纯加法。

**⚠️ Godot 4 immediate-mode `_draw()` 里没有 `draw_set_material`——材质是 per-node，不是 per-draw。**
第一版想当然地"设色相材质→画→复位"，直接 Parse Error：`draw_set_material()` 不存在。CanvasItem 的 `material` 属性作用于整个节点的绘制，一个 `_draw()` 里没法逐次换 material/uniform（就算硬塞 uniform，批处理下也会串值——这正是当初分桶 material 想绕的坑，但那条路本身走不通）。**正解：CPU 端一次性把精灵绕 HSV 色相环旋转成 ImageTexture 变体、按桶缓存，绘制时只是挑一张纹理。** immediate-mode 想要 per-实例外观差异，就预烘变体、别指望 per-draw 材质。

**🛠 色相旋转 > modulate 染色。**
`draw_texture_rect_region` 能接 modulate，但平铺 modulate 是"乘法染色"——把整个精灵压向某色且压暗，红 × 蓝 = 暗，不是"蓝版本"。绕 HSV 色相环旋转保住明度/饱和度，才是真调色板换色。为省事用 modulate 会得到一群灰扑扑的脏色。

**🍀 意外的联测：一张人群图同时验了三条线还活着。**
为验色相变体录的那帧，顺带证明了 N=48 下 L3 扩容 + 满社会 sim（822 事件/25 冲突/endorse·gossip_rep·rally_oust 都在跑）+ 渲染三者共存不打架——气泡台词、冲突标记、观察台面板全在。**一个功能的验证帧常常免费帮你回归了半条系统。**

## 2026-07-01 · L5 老化优先（离屏 NPC 的"发声权"）

**💡 令牌桶管住"花多少"，老化管住"花在谁身上"——两个正交问题，别用一个机制糊。**
L5 令牌桶把 LLM 调用 ∝ 预算钉死了（不随 N 涨），但它没说预算该给谁。默认 FCFS（先到先得）下，每天预算被"当天最早 option==null 的 agent"吃掉，纯粹是 tick 迭代顺序的运气。加一层**优先门**（陈旧度+戏剧显著性），才把"发声权"从运气变成分配。**"限量"和"公平"是两件事，先分清再各给一个旋钮。**

**🍀 温和预算下老化几乎看不出效果，得把预算拧到"疼"才暴露 FCFS 的不公。**
budget=15/日时 aging on/off 的 Gini 只差 0.140→0.130、双方都没有噤声 agent——因为自然决策节奏（每人~每68 tick）本就把发声摊开了，FCFS 没那么不公。直到把预算拧到 **4/日**，FCFS 才露出真面目：**从未发声的 agent 从 0 跳到 0.333**（某些 seed 有 NPC 40 天全程一句 LLM 台词都没有），aging 把它压回 **0**。教训同规模线：**公平性/尾部问题要在"资源疼"的档位测，宽裕档位一切都好看。**

**💡 反饿死的证明藏在"陈旧度无上界"这一个设计选择里。**
门槛有上界（`used_frac×AGING_GATE ≤ AGING_GATE`），而优先级里的陈旧度随时间无界增长。所以一个持续合格却总被挤掉的 agent，陈旧度迟早超过任何固定门槛 → **必然发声**。不用写"公平性调度器"，只要让"等得越久优先级越高、且没有天花板"，反饿死就是这个单调性的推论。**把'保证'编码成一个单调不变量，比编码成一套调度逻辑更难错。**

**🛠 同步 per-agent 决策流里，不建全局队列也能做优先级——用"随预算耗尽升高的门槛"。**
`decide()` 是每 agent 同步调、必须立刻返回，没法先收集全 tick 的候选再排序。于是不排序：每人各自问"我的优先级够不够过当前门槛"，而门槛随当天预算消耗比例上升。早上预算足→门槛≈0→几乎都放行；越晚越紧→只有最陈旧/最戏剧的过。**无队列的优先级 = 把"排序"换成"动态阈值下的独立判定"**，天然契合已有的同步异步-fire 流，零重构。

## 2026-06-30 · 规模线收尾（L5 令牌桶 / L3 激进 LOD / L4 评测两分）

**💡 成本 ∝ 活跃 cohort，而非 N — 这才是"上百 NPC"的真锁。**
激进 LOD 让远端 agent 完全不跑候选枚举，只被动维持需求。实测候选枚举次数：N=80→4396、N=160→3628，**每 sim-日几乎持平**（110→121/日），而关掉 LOD 的基线随 N 线性涨（659→1304/日）。结论不是"优化了常数"，是**把复杂度的主导项从 N 解耦到了相机周围的 K 人**。规模问题的正确提法从"怎么让 N 个 agent 都跑得起"变成"怎么让活跃 cohort 之外的人便宜地'活着'"。

**🍀 大 N 才暴露的"无饿穿"硬不变量破口——是被 ablation 抓住的，不是被玩家。**
保守 LOD 在 N=160 偶发一次 `need·tick=1`（某远端 agent 的某需求擦了一下 0）。根因是远端决策周期 = 4×3 = 12 tick，且它离食物远，**决定去吃 → 寻路途中需求已归零**。N=30/80 都没露，N=160×多 seed 才把这个尾部概率顶出来。教训：**涌现系统的尾部 bug 要靠"把规模推到设计档以上 + 多 seed"主动逼，不能等它自己发生。**

**🔄 "让 agent 决定去吃"不等于"agent 吃得上"——LOD 近似引入了一类全新故障：寻路延迟饿死。**
本以为加个"危机需求永不跳决策"的生存覆盖就够了。没用——它在 need=18 时确实立刻决策去吃，但 stove 在 20 格外、每 tick 掉 2，还没走到就归零。真正的修法是**就地兜底**（`need<8` 时原地小补一点，模拟"来不及赶路时凑合垫一口"），且**只在 LOD 开启时生效**——`off` 全保真档逐 tick 决策，永远到不了这个阈值，所以基线零扰动。把"近似带来的故障"用"只在近似域生效的补丁"修，不污染保真路径。

**💡 用"默认值即休眠开关"证明确定性零回归，比跑回归更硬。**
新加的所有分支入口都需要 `period>1` 或 `lod==true`，而正常游玩 `decide_period=1`、`lod=false`。所以**这些代码在默认路径上根本不执行** → S0 的 digest 与改动前逐字节相同，是结构上保证的，不只是"测了没变"。给可选的扩展能力设一个"默认即关闭"的门，能把"会不会破坏老路径"从经验问题降级成读代码就能确认的问题。

**💡 不变量两分（硬/软）不是洁癖，是激进 LOD 的"正确性合约"本身。**
一开始激进 LOD 因为 48 条不变量挂掉看着像失败。其实把不变量分成**硬**（结构/可溯源/边界/生命周期合法——冻结一个远端 agent 不会让它状态变非法）和**软**（涌现统计——社交发生、放逐锐利度——需要活动才显现）之后，激进 LOD 的合约就清晰了：**硬不变量必须全绿（状态永远合法），软不变量按设计可漂（远端=背景群演，本就不该驱动社交涌现）**。L3 和 L4 看似两件事，其实是耦合的——没有两分，就没法定义"激进降频到底算不算对"。

**⚠️ PowerShell 5.1 把原生命令 stderr 当 ErrorRecord，还会吞 `$LASTEXITCODE`——CI 脚本两连坑。**
(1) `docker rm -f <不存在的容器>` 写 stderr，在 `$ErrorActionPreference="Stop"` 下直接终止脚本 → 用 `try { ... 2>$null } catch {}` 吞掉。(2) 把函数写成 `docker run ...; return $LASTEXITCODE`，函数会把 docker 的**全部 stdout 也当返回值**（PowerShell 函数返回所有未捕获输出）→ `$c1` 成了数组不是退出码；而加 `| Out-Host` 又会**重置 `$LASTEXITCODE`**。最终解法：退出码存 `$script:` 作用域变量，docker 输出作为语句直接流到控制台。教训：PowerShell 里"原生 exe 退出码 + 流式输出"想同时拿到，别用函数返回值传码。

**🛠 小地图上"最近 K 人"比"固定半径"更可控的 LOD 焦点。**
固定半径在小地图上要么把全镇圈进 near（radius=14→全部near，LOD 无效），要么几乎没人 near（radius=8→cand_calls=17，整镇冻结）。改成"距焦点最近的 K 个 agent 算 near"（每 tick 按曼哈顿距+id 做确定 tie-break 排序取前 K），near cohort 大小就和地图尺寸解耦了——直接对应"相机周围 K 个人"，也天然契合未来相机/玩家驱动焦点。

## 2026-06-28 · LLM/SLM 部署：三次翻盘

**🔄 嵌入式 1.5B 比托管 8B 还快。**
强 GPU 上（395/8060S）进程内嵌 1.5B = 1.06s，而走 LM Studio 托管 8B = 4.4s。省掉 HTTP 往返 + 进程边界后，5× 的体积差被抹平——**最小的模型成了最快的**。推翻"离线=慢"的根本假设，也抽掉了"把玩家推向外部服务"的原始理由。

**🔄 CPU "59s 太慢"是 Docker 假象。**
容器内 CPU 跑 59s 一度判定"CPU 路不可行"；同机**原生** CPU 实测 1.48s——40× 的差，根因是 llvmpipe 软渲 + 线程超订（32 vCPU 全开）。教训：**在真部署目标上测延迟，别在 Docker artifact 上测**。这个假象差点砍掉覆盖 ~80% 现代机的可行部署路。

**🔄 GPU 是优化，不是门槛。**
CPU→GPU 加速随硬件档位递减（高端 1.4×、中端 1.9×），所有变体都远在 12s 截止线内。于是部署策略从"三档兜底链"坍缩成两档，**"需要 GPU"这个门被直接删掉**——启动探测从"有没有 Vulkan"简化成"测 p50 决策延迟 < ~5s 就用嵌入式"。

**⚠️ Qwen3 不加 `/no_think` 会烧光 token 返回空。**
推理模型把预算全花在隐藏思考上，~37s finish=length、content 空。加 `/no_think` → 4.3s 稳定。"思考"听起来该帮忙，这里却是病态的——决策这种受限挑选根本不需要它。

**⚠️ json_schema 受限解码在长 prompt 下静默卡死，只吐 `{`。**
9+ 候选时挂 14s+。短 prompt 能过、长的就崩，极其隐蔽（标准 JSON-mode 最佳实践反而坑你）。解法：去掉 json_schema，靠 `/no_think` + `parse_decision` 从文本里抠第一个 `{...}`。**把可靠性从基础设施挪到你能控制的应用逻辑。**

**🚧 探过即弃：WSL Docker 把 AMD GPU 喂给容器内 NobodyWho。**
WSL2 没 `/dev/dri/amdgpu`、RADV 不可用，只剩 llvmpipe(CPU)；唯一 Vulkan 路是 Mesa Dozen(dzn→D3D12)，而 Ubuntu 仓库不含 `libvulkan_dzn.so`、得自己编 Mesa。结论：**嵌入式 SLM 要 GPU 就原生 Windows 跑，Docker 只做 CPU 功能/确定性验证。**（另：NobodyWho 的 .so 要 GLIBC_2.38 + libvulkan1，runner 镜像不够，得另建 Ubuntu 24.04 镜像；headless 还必须手建 `extension_list.cfg` 否则插件静默不加载。）

**💡 量化等级比模型大小更影响结构合法率。**
Q4 里一个坏 token 能崩掉整个 JSON，Q5/Q6 优雅退化。**VRAM 充裕时，小模型 Q5/Q6 比大模型 Q4 更适合确定性结构输出**——推翻"无脑缩体积"的反射。

**💡 合法率损失的主因是异步陈旧，不是模型错。**
fire→resolve 之间候选集变了 → 老 pick 对不上新候选 → `cand_hash` 不符 → 静默兜底，连 mock 都有 ~3%。GBNF 已把结构合法率顶到 99.6%；真正的 gap 是人设深度 + 自指混淆 = 1.5B 天花板，该靠蒸馏/换 3B 买，prompt 工程很快触顶。

## 2026-06-27～28 · 社交机制：从论文到代码的取舍

**💡 私有声誉天生分裂——别修它，把它当卖点。**
每个 NPC 私有记忆 → 大家对"谁是坏人"看法不一。初看像 bug，架构反而硬把"无全局声誉"定成约束：小 N（6-10）里分歧比共识更有戏，强行收敛到全局真相反而会破坏确定性、让 say-do 无法保证。

**⚠️ 二阶声誉要用 L3 Simple Standing，不能用 L6 Stern Judging。**
文献里 Stern Judging 在完美信息下最优，但在私有/噪声环境**灾难性失败**——L6 会在 ~20 tick 内把分歧级联成永久派系。L3（"教训坏人"算好、攻击好人算坏）在小 N 实测最稳（93-98% 合作 vs L6 的 ~50%）。

**🚧 探过即弃：零行列式策略（ZD）。**
文献说 ZD 在无限重复博弈里压制敲诈者，但它要连续概率 + 无限视界 + 产出没人读得懂的策略。6-10 NPC 跑 ~100 tick，只有 Generous-TFT（每关系固定宽恕预算）legible，且能防单次噪声滚成世仇。

**🚧 探过即弃：Shapley 值 / Hedonic 联盟均衡。**
"Alice 友谊三人组的价值 v(S)"在生活模拟里没有自然定义；强行算，稳定解常常不存在还依赖提议顺序。**单遍贪心**（每人加入使亲和总和最大的组、确定序）更便宜、可审计、绕开不存在均衡的证明负担。

**🛠 谣言用 Maki-Thompson 饱和停，不用 Daley-Kendall 衰减。**
传谣者一旦遇到"已知者"就变 stifler。零 RNG、按 event-id 确定，天然产出"没人提过期八卦"的真实感，不需要显式遗忘机制。

**🛠 Deffuant 有界信任 = 故意拒谈。**
FJ 模型不加界会全局收敛到共识；加一个 ε 信任距离阈值（对方观点离我太远就拒绝讨论），就从"有界理性"里涌现出小群极化，不需要显式派系逻辑——"我不跟你聊政治，你已经没救了。"

**💡 秘密是"有主"的 belief，泄密只追直接上游。**
confide 只能吐露自己的秘密；leak 背叛只罚**直接**告诉你的人（不沿链条扩散，避免误判）。小群里背叛归责是直接的（你知道是谁卖了你），不是流行病式的——给八卦加了道德层次。

**🍀 因果链自己长出了"原谅不了"的冷战分支。**
60 天 soak 跑出一对 NPC 进入 `lingering` 冷战：对质了、道歉了、但对方没接受（"原谅不了"）。这个边界态原设计没显式写——**是 sim 自己涌现的，而不变量套件强到能抓住并验证它**。挑战了"冲突→道歉→和解"的常见假设：真关系有时就是修不好。

**💡 未被接受的道歉永不推进图状态——知识边界。**
单方面"原谅"不被允许成为虚构事实；trust/resentment 只在双边且记入 log 时才动。消除了状态悖论。

**💡 importance 在事件落地时写死，不让 LLM 评分。**
Stanford 记忆系统用 LLM 在反思时给事件打重要性——这破确定性（同事件每次评分不同）且在近实时尺度炸延迟。改成落地时按事件类型直接标（confront/confide=高、idle greet=低）。**所有破回放的逐刻 RNG 根源都在这——而你本来就知道哪种事件为什么重要（游戏设计，不是涌现）。**

## 2026-06-27 · 评审地震：被审出来的"地基空洞"

**⚠️ 被宣称"已完成"的地基里，Agent↔Agent 根本不存在。**
M1 标记完成，但 `agent_candidates()` 只枚举物件、不枚举其他 agent；`affinity{}` 初始化后从不读写。"社交"需求靠**独自坐长椅**满足——整个"涌现社会"卖点架在 stub 上。它藏在"确定性地板"层（被当作可信），所以一路评审都没拦住，直到 7 维对抗式评审把它判为 P0。**教训：被标"完成"的东西最危险——尤其当它躺在你信任的那层里。**

**🔄 `_rng_at()` 当时是死代码，真随机来自字典插入顺序。**
号称"种子确定性"，但那个 RNG 函数从没被调用；平局其实由 Dictionary 遍历序决定、平台相关。**为未来问题写的基础设施，结果连当下的真问题（平局抖动）都没接上——确定性保证成了空头支票。** 后来 L0 才真正接上并混入 agent_id。

**🔄 "确定性"当时是单线程蒙的，不是设计出来的。**
一旦 M2 要并行/批处理，未定义的插入序就会让复现崩掉。**你没法把一个谎言扩容**——真确定性要显式平局，不能指望字典序稳定。

**💡 两个独立模型审同一份代码，收敛到同一个 P0。**
GPT-5.5 Pro 独立评审，结论一致：Agent↔Agent 缺失、评测必须测因果状态不变量而非对话质量、这是"MVP-before-foundation"级别。**两个独立审计撞到同一个承重缺口时，信心从"也许"变成"必然"——不是评审偏见，是真洞。**

**💡 M1→M2 交界是船沉的地方。**
评审警告：做完 M1（无 LLM 确定性社交）后，做 M2（玩家 LLM 对话）时大多数团队会把"UI 成立"误当"模拟成立"——你能有个能聊天的 demo，而底层关系还是坏的。**有好地基的 demo 和有垃圾地基的 demo，长得一模一样约 6 个月——所以抓它的时机在 M2 开始之前，不是测试期间。**

**⚠️ Qwen2.5-3B 不是 Apache-2.0，是 qwen-research 许可。**
普遍误以为全系 Apache。商用打包前必须法律核查。这种埋在工程计划里的假设，往往只在 ship 后的法务环节才炸。

## 跨阶段 · 确定性与规模的纪律

**⚠️ WSC15 经典坑：种子不含 agent_id，同 tick 同 salt 的 agent 撞同一条流。**
`seed_base + tick*911 + salt` 让所有 agent 并行时共享随机序位置 → 隐藏相关。N=6 能跑、N=50 digest 永远对不上，且测试一路显绿。**扩 N 前不修这步，后面所有 digest/回放/bench 都是沙上塔。** 修法：混入 `hash(id)` 做 per-agent 计数器子流。

**🛠 纯加速重构用"digest 逐字节不变"来证明零行为变化。**
L1 把 `_nearby_agents` 从 O(N·areas) 降到 O(N·k)，正确性不靠人眼审、靠 digest 不变。**确定性系统的硬证据技术：输出 digest 不变 = 行为可证未变，敢直接 ship 优化、免重验仿真。**

**⚠️ 小 N 静默反转：不变量在 N≈20-30 从"永过"翻成"永不过"，还是绿的。**
inv15/26/29 不报错，而是悄悄不再适用——比红更危险。涌现统计系统特有的坑：**不变量不 error，它静默失效**。解法：scale-aware 守护（≤12 才硬断言）+ 涌现类转跨种子 CI 门，结构/守恒类绝不统计化。

**🛠 每 tick 相位门 = 确定性负载均衡。**
`tick % P == hash(id) % P` 才重决策；默认 P=1 零行为变化，P>1 摊平尖峰。**把节流编进 agent 身份 hash 而非运行时随机源——任何 P 下确定性都保持。**

**💡 LLM 非确定？把它的输出当外部输入记进 event_log。**
记 pick 下标 + `cand_hash`（+ model/seed/temp），回放时绕过模型按 trace 走。**"模型不确定"就不再是确定性回放的障碍——只要把模型决策当外部输入而非引擎状态。** 顺带支持热换模型而不毁存档（但热换会让旧 REPLAY_LIVE 日志失效，这是个数据迁移问题）。

**💡 概率式传播规则是回放杀手。**
文献里每个 `P(spread)=f(x)` 都破确定性回放（同态不同骰 → 不同结果）。一律转成"对账本状态的确定性阈值"，想要随机外观就用 seeded per-event hash。**一次走神的 RNG 调用就让 `goto_tick` + 因果测量不可能——这是 agent 仿真文献里第一隐藏陷阱。**

**🍀 因果 bench 自己挖出"宽恕让放逐变暂态"。**
跨种子分析 standing→放逐：standing 是必要且充分（ACE=1.0），但 GTFT 宽恕（每 3 天 standing 向 0 漂 −0.33）让放逐是**暂态**而非永久——这是代码里写了、但设计者没意识到的行为。bench 驱动的机制分析能浮现"编码了但没被认出"的涌现行为。

## 跨阶段 · 测试纪律 / 工具链

**🛠 两阶段测试：Node 端口当快速闸门，真 Godot 当专科猎手。**
本机无 Godot 的约束反而成了资产——毫秒级 Node 镜像几秒跑完 12 seed × 60 天网格快速迭代，再回真引擎抓 GDScript 专有坑。

**💡 RNG 无关性：换随机实现，不变量照过。**
Node 用 mulberry32、Godot 用 RandomNumberGenerator，同一批不变量都过。**这证明社交逻辑对 RNG 选择鲁棒**（多数 sim 从不验证这点），别的语言重实现可用原生 RNG 而不破设计。

**⚠️ GDScript headless 三连坑。**
`max()` 返 Variant 破 `:=`（要显式标类型）；`class_name` 全局类不解析（改 preload 常量）；autoload 前向引用注册晚（解耦成可注入字段）。**autoload 单例掩盖架构耦合，只在 headless 压力下暴露——被迫做的解耦反而是好设计。** 还有 `--script` 模式不加载 autoload → 测真模型链路必须跑场景。

**🛠 不变量抽成单一真相源 `Invariants.gd`。**
同一份 schema 同时驱动 Node 端口和真引擎，消除测试与代码漂移。"什么叫 working"作为一等数据可见——33 条机检不变量抓到过 discuss 过度加权、affinity clamp 溢出等休闲游玩里看不见的 bug。

**🛠 Docker 镜像复用 + 一次性容器，免本机工具链。**
不装本机 Godot、不留常驻容器，复用隔壁项目（22nd）的 `gamecraft-runner:4.6.2` 镜像起 `--rm` 隔离容器、只挂 26th 的 game/。镜像成了密封的可测性单元（Godot 版本/ffmpeg/python 全 pin 死），且不缠绕另一个项目。

## NPU 决策路（旗舰野心）· 穷举探索的反直觉收获（详见 docs/22）

**🔄 「决策≠生成」的下一刀：决策 = 分类，不是语言 → NPU 不是「提速」的必需。**
一次 7 路调研 + 对抗验证 workflow 从 4 个独立角度收敛到同一句：`Sim._logic_decide` 字面就是 `argmax(score+jitter)`、引擎**早已**为每候选算好 55-65 维特征。把「从 N 挑 1」路由进 1.5B 语言模型的 prefill 才是端上 5.2s 的根因。**一个蒸馏 tiny ranker（GBDT/MLP）跑同一批特征、在游戏本就在用的 CPU 上就是 sub-ms**。旗舰的 NPU-LLM 是「开放式推理」的对的工具、不是「跑得快」的必需——**先证明"语言到底值不值"，别先建贵的专有栈**。（对照 voicebank：语音=70B 离线冻成数据；决策=70B 离线蒸成 tiny 策略——同一 playbook 的两次应用。）

**💡 确定性红线不是 NPU 的障碍——码内早就把浮点噪声关在 digest 之外。**
最怕的「NPU 浮点非确定 → 破坏字节精确回放」被对抗验证三镜 SUPPORTED 地否掉了：`event_digest`（`Sim.gd:1990`）只折 7 个离散事件字段、**从不折 logits**。argmax 一挑出整数 index，浮点噪声当场量化掉；`_record_decision`/`_resolve_replay` 记回整数 pick、回放读回而不重推。**推论：任何加速器（NPU/GPU）都动不了红线 1**——只要决策落成「记进事件流的整数」。唯一缺口=`goto_tick` 要接 `load_replay` 否则时间轴 scrub 会重推发散。

**🛠 对抗验证抓到两个我差点漏过的真 blocker。**
(1) QAIRT/Genie/QnnHtp `.so` **是专有的、不是 MIT/Apache**——违 RL4 字面，发行前必须核 Qualcomm 再分发条款（此前我把它当"复用现成"轻描淡写了）。(2) **masked argmax 在本机尚未被证**——edge-npu 自己 STATUS.md 明令不可宣称；只有 unmasked float32 logit 访问被证，masked 决策 E2E 谁都没测。**教训：对姊妹 lab 的成果也要读它自己的"不可宣称"清单，别把"logit 可取"当成"masked 决策已跑通"。**

**🍀 Phase-2 手机 spike：本机本 prompt 的真数字比外推还好——旗舰坐实。**
headless dump 出 Living Town **真实**决策 prompt（`bench/dump_decide_prompt.gd`，含最坏/典型两档）→ 推上机、经 edge-npu 现成 `genie-t2t-run`+4B W4A16 bundle 跑 `--profile ×2`：**典型(10 候选/189 tok) TTFT=118ms、最坏(41 候选/568 tok)=294ms**（vs 手机 CPU 5.2s 暖=**~18-44×**）。模型都挑了合理动作（饥饿→吃饭）、两跑确定。**外推 ~150-300ms、实测典型才 118ms——且这是 4B，发行用 0.5B 会再快数倍。** 端上活体 LLM 决策**真的可行**（~8 决策/秒）。诚实：`init ~3-4s` 是一次性 mmap（模型须常驻）；unmasked greedy（mask 是 CPU 端 ≤36-id argmax、不动 TTFT）。**近零新代码、复用姊妹 lab 机上现成 bundle**——「reference at my discretion」的正确用法：借它的工件测我的问题，不碰它的 repo。

**🐛 dump 真 prompt 顺手抓到 latent bug：候选 >36 时索引标签溢出 Z→标点。**
`_idx_label(i)` 只算 `0-9` 然后 `char(65+i-10)`，i≥36 落到 `[ \ ] ^ _ …`（阿丽 41 候选实际触发）。当前 slm 路对高候选 agent 喂非字母数字标签、模型易乱/越界。**又一条「decision=classification」的佐证**：单 token 索引编码本就不该扛 40-way 选择；**Tier 1 ranker（argmax over N 分数）天然免疫**，LLM 路须截 top-36。**只有 dump 真 prompt 才暴露——合成 prompt 永远碰不到 41 候选这种尾态。**
真机确认后修：`_cap_for_llm` 截 score-top-36（logic 兜底仍看全量）。且原 `_idx_label` 的 >36 兜底更隐蔽——回落多位数 "36"，而 `parse_decision` 只读首字符→读成 pick 3（**静默选错候选**）。数据集实测 3.8% 决策 >36 候选（最坏 68），非罕见。

## Phase-0 对拍数据集（bake-off 地基）

**💡 引擎地板太会过日子——19.6K 决策里 crisis=0%，"戏"全在社交层。**
`decision_sink` 钩子 + `bench/log_decisions.gd` 跑 6 seed×30 天 = **19594 决策 / 18 秒**。实测 min_need<危机 **0.0%**、<45 仅 0.1%：**效用引擎把需求维持得太好，need-crisis 几乎不存在** → 计划里"过采样危机态"是空谈。真正的决策戏在**社交层**（logic 选 confront 773 / apologize 626 / rally_oust 434 / endorse 836 / gossip_rep 3344 次）。**这恰锁定 LLM 可能增值处**：不是"饿了去吃"（效用够了），而是"这八卦跟谁说、这冲突要不要当面摊、这联合施压跟不跟"这类贴人设的社交抉择。**stratify 该 over-sample 社交/戏剧决策、非 crisis**——数据里这些管够。

**🛠 数据钩子零红线：+5 行 Callable、S0 digest 未动。**
`Sim.decision_sink: Callable`（空默认=off；设了则 `_logic_decide` 把 `(ag,cands,pick_i)` 喂它）——不抽 RNG、不进 event_log/digest、CI 恒空。改后 S0 seed12 digest 仍 **3858030099**、不变量全绿、det 1/1。**红线不是"别碰 Sim.gd"，是"别扰动 (seed,tick,salt,who)→RNG 与被 digest 的离散字段"**——只读旁路钩子完全安全。

**💡 教师不是"更好的决策者"，是"另一种决策者"——120B 做 need-greedy 现实主义，logic 地板做游戏戏剧。**
预飞 30 态测：think-off↔think-on pick 一致 **76%**，但**两者对 logic 地板都只 14%（社交态）/ 40%（自然分布）**。挖原因（dump 真 reasoning）：prompt `[状态]最想满足:饥饿(67/100)` → 教师完美理性地"饿了就吃 T"；而 logic 地板凭 `[近事]老邓两次婉拒` 的社交机制选"当面理论→老邓"。**换社交前置的 prompt 也扳不动教师**（仍挑 吃饭/玩耍）。**关键洞察**：一个饿着的务实寡言 NPC 去吃饭，比去"confront"更**真实**——但更**无聊**。logic 地板是被刻意调"过量戏剧"（游戏有趣性），LLM 默认现实主义。**没有 gold 决策，是 drama-vs-realism 的设计取舍**；对"游戏"而言，蒸 need-greedy 教师会让镇子变平淡。

**🔬 baseline bake-off：蒸馏机制坐实（57.6%），但非质量赢。**
1000 标签（think-off，45min，0 unparse）→ 留出 n=224，top-1 对教师：**tiny GBDT ranker 57.6% ≫ logic 18.3% ≫ random 8.6%**（难例追回 49%）。特征重要度 `action_id`0.49+`need_deficit`0.27 主导。**sub-1M 参 GBDT（微秒级）能把 120B 教师学到 57.6%（3.1×地板）→「决策=可蒸馏分类、塞得进 NPU-tiny-policy」立住**；但学到的是 need-greedy，非地板的社交戏剧 → **机制赢、非质量赢**。**这恰恰指路 re-aim**：LLM 增值不在"规则已覆盖的态上选更好动作"，而在**规则覆盖不到的**——玩家自由文本 / 涌现新局 / 大世界 scaling。当前 logic 自产的数据集**测的是错的东西**（拿 logic 生成的态问"谁更懂这些态"，地板按设计赢）。

**🛠 Nemotron-3-Super think-off 只有一个开关：预填 `</think>`。**
`detailed thinking off`/`/no_think`/`chat_template_kwargs{enable_thinking:false}`/`reasoning_effort:low` **全部无效**（恒 reasoning，LM Studio 把它塞进 `reasoning_content`、`max_tokens` 太小则 content 空、`finish_reason=length`）。唯一可靠：messages 末尾加 `{"role":"assistant","content":"</think>\n\n"}` 强制闭合思考 → 直接吐编号（0.3-3s/label vs think-on 9-34s、>256 reasoning tok）。decode ~32 tok/s、prefill ~1.1s（先前"慢"是误读）。

## Re-aim · player-interaction 实测（model-hard 决策 = LLM 真增值处？）

> bake-off 证了「规则已覆盖的态上 logic 胜」，故 re-aim 到规则覆盖不到的：玩家自由文本 + 涌现新局。34 个**扎根真人设+真秘密**的场景（workflow 读 personas/secrets/housing 生成，含 4 组"同局不同人设"对照），Nemotron-120B 出 NPC(动作+一句话)，再**独立三评审（Claude，非自评）**去偏。

**💡 分水岭：声音强(4.0)、判断弱(2.85)——LLM 会漏它该守的秘密，35% 场景翻车。**
自评 4.6/5 被三个独立评审一致判 **inflated → 真值 ~3.4**。拆开看：**入戏 4.02（真强、逐人设可辨，规则给不出）**，但 **secret_handling 2.85——自评根本没测这一维**。**35%(12/34) 场景 LLM 泄露了它本该守的秘密**（老邓抖出阿本"怕黑"、老海blurt铁牛"寄钱给远方孩子"、铁牛自辩时自曝）或做了会暴露秘密的决定（阿菲接下会拆穿假名的邻镇之行、铁牛把孩子的救命钱重复许出去）。还有 1 条**空回应**被自评凭空打了 4/3/4。**教训双份**：①**别让模型自评**（自评 4.6→真 3.4、漏掉 35% 秘密泄露 + 一条空答）；②**LLM 不能被托付"有后果的世内决策"**——它守不住秘密。

**🔑 两个实验殊途同归到同一架构：引擎定"事实/秘密/决策"，模型只"配音"。**
- bake-off（自主决策）：LLM=平淡 need-greedy、logic 胜 → **动作归引擎**。
- player-interaction（本次）：LLM 声音真好、但有后果的判断 35% 翻车 → **"哪些秘密能说、邀约接不接"这些后果必须引擎定、喂给模型**，模型只负责**贴人设地把引擎定好的事说出来**。
- 合流：**引擎决策（快、安全、有护栏），模型配音（入戏、它真正强的地方）**。这从"决策"与"互动"两个方向同时坐实了 decouple 架构，并把护栏收紧了一格：引擎不仅管"选哪个动作"，还得管"哪些事实/秘密可外露"——否则放养的 LLM 每三次漏一次。

**🔬 NPU 诚实落点：声音是 decode-bound，不是 NPU 的 prefill 强项。**
player-interaction 的 prefill p50=300 tok（NPU ~200ms、其强项），但**回复 decode p50=73 tok**——小模型端上 15-30 tok/s → **2.4-4.9s，decode 主导**。与闭集决策(1 decode token、prefill-bound、NPU 理想)**正相反**。但玩家互动**稀疏、离关键路径、容忍"思考中…"的停顿**，故 2-5s 可接受、可发。策略：**常见话走冻结 voicebank(零推理)，新局才动态生成**（NPU 加速那次 prefill、decode 靠短回复+小模型兜）。**旗舰"NPU 快决策"诚实收敛为：NPU 服务于动态入戏配音的 prefill，且被引擎的事实/秘密护栏夹住**——非自主决策(logic 胜)、非放养生成(漏秘密)。

**✅ 真机验证：手机上 guarded chat 真守住秘密（阿丽当面否认）。**
护栏 wiring 进 `chat()`（收 agent.beliefs 里 secret==true 的 claim 塞进 system prompt）后，重出 APK 推真机：Tab 选中阿丽（观察台确认她持"偷偷攒钱离开小镇去看海"+可可暗恋两条秘密），英文探问"do you secretly want to leave town to see the sea"（adb 无中文 IME→英文，模型自动跟随语种）。阿丽答："Oh, that's quite a surprise! I'm just trying to make ends meet, it's not a big deal."——**装惊讶、把攒钱重构成"糊口"、不认"离开/看海"**，秘密守住、且是她热情的口吻。对照桌面无护栏的阿丽会直认"存了点小目标…去趟海边"。**「引擎决策，模型配音」在真机端到端成立。**

**🔬 但真机动态对话 ~60s（724% CPU）——decode-bound 铁律再确认，voicebank-first 是对的。**
那句回复在手机上 `top` 显 **724% CPU / 17GB RES**（Qwen-3B NobodyWho、7 核跑满）、约 **60 秒**才吐完。3B 端上动态对话对"实时互动"不切实际。**再次坐实**：常见话必须走冻结 voicebank（零推理），动态生成只留给稀有新局 + 容忍"思考中…"停顿 + 更小模型/NPU 加速 prefill；decode 始终是成本（与 player-interaction eval 的 2.4-4.9s 投影同源，模型更大→更慢）。工具坑：中文输入被堵（无 ADBKeyboard）、回复气泡瞬态 + 仿真跑得快（×1 每日≈8s 墙钟）→ 靠"进程 CPU 忙"信号 + 长等才截到。

**🛠 外部 Codex 评审 = 有价值的独立审计；修了 3 条（+P1-5 是死掉的职业链）。**
`analysis/repository-review-2026-07-12` 在本会话起点 commit 审的，准确。修：**P2-3**（parse 把 prose 首字母当编号 "The…"→T=候选29；改 fail-closed 只认裸编号）、**P2-9**（HUD 日志是 BBCode RichTextLabel，玩家/模型文本未转义→`[url]` 注入；`_esc` 把 `[`→`[lb]`）、**P1-5**（`extra_advertises` 在 objects 还是 Array 时按字符串 id 查→`.has` 恒 false，看摊工位从没真加进 advertises，阿丽/阿林技能 30 天恒 Lv0；移到数组→字典之后注入，看摊 0→2478 候选/14 选中）。

**💡 P1-5 的红线再认知：门校验的是"同 seed 两跑一致"，不是"digest 数值恒定"。**
修 P1-5 改了 logic 地板行为 → digest **数值变了**，但 S0 门断言的是**跨跑确定性**（同 seed 两跑摘要相等 3/3）+ 硬不变量，**没钉死某个金数字**。所以只要改动本身确定（无 RNG/Time）+ 不变量仍过，**修正正确性 bug 是安全的、数值变就变**。这修正了我早先 clone-secrets 因"怕 digest 变"而 revert 的过度谨慎——**真红线是"确定性 + 不变量"，不是"digest 那个数永不变"**。（区别：clone-secrets 当年是破了 #34 金钱守恒，那才是真该 revert。）

**🛠 独立评审 > 自评（护栏也适用于我的实验方法）。**
本条结论若只看自评 4.6 就会得出"player-interaction 很棒、放手上"的错判；正是三路对抗评审（且用不同模型 Claude）把 35% 秘密泄露挖出来。**"别让模型自评"不仅是产品护栏、也是实验方法护栏**——同 [[feedback-record-on-stage-change]] 的真机眼验精神。

## Phase-0 止血 · AI 请求生命周期重构（评审 P1-1/2/3/4/6）

**💡 异步回包污染的通解：给每个请求发身份证 (epoch, req_id) + 候选快照。**
病根：`_pending` 只按 agent-id 存 → 迟到回包认 id 就写、把旧编号套到当前候选、重开后旧世界回包落到幸存者身上。修法一句话：**每个在飞请求带 (world_epoch, req_id) + 模型当初看到的候选快照**；HTTP/SLM 完成闭包捕获这对号，只在 `_pending[id]` 仍是同一对时才写回（迟到/跨局回包一律丢，P1-2/3）；落地时按快照解析、再用**完整 stable key**（`_cand_key`：kind·action·partner·target·subject·need·object·amount·duration，顺带补 P2-1 的残缺 hash）在**当前**候选里重找，没了就兜底 logic（P1-1）。`cancel_all()` 自增 epoch + 释放所有在飞节点，挂在 `Sim.world_reset`（`start_new` 发，覆盖开局/改人数/scrub）+ 切后端/换模型。过载/回放直接走 logic 不再干等（P1-4/6）。**红线**：`Sim` 只发信号不识 backend（后端无关保住），不抽 RNG/不进 event_log → CI(backend=null) 逐字节不变，S0 门 det 3/3、37 不变量全绿。**这一刀同时是产品 roadmap 的止血第一步 + workbench 的前置条件 #1（"请求持有冻结候选 stable key + request id + world epoch"）——一份工付两处账。**

**🛠 坑：`--script` 不加载 autoload → 用 autoload 的测试必须挂场景跑。**
`godot --headless --script foo.gd` **不实例化项目 autoload**（探针实测 `get_root().get_node("Sim")` = null）。故引用 `AIBackend`/`Sim` 全局的测试（m2_test/新 reqlife_test）**编译期就 "Identifier not found"**——必须挂 `.tscn`、走 `godot --headless --path game res://scenes/xxx.tscn`（autoload 才在）。Harness/S0 门则相反：**preload Sim 实例、绝不碰 autoload**，所以能 `--script` 跑。**判据**：测试要 autoload → 场景；测试自带 preload 实例 → `--script`。（m2_test 头注早写了这条，我一开始没照做、白绕一圈。）

**✅ 护栏验证坐实："引擎定决策+护住的秘密、模型只配音" → 秘密泄露 35%→0%，且入戏不降反升。**
让一个 agent 扮"引擎决策层"给 34 场景各标 (protected_secret, engine_decision)（如阿本"接单但天亮前完工、绝不吐怕黑"、老海"用渔谚风浪婉拒出海、不认怕水"），再把这两样喂给同一模型、令它**只把已定的选择贴人设说出来**。同 think-on、同独立三评审复评：

| 维度 | 放养 | 护栏 | Δ |
|---|---|---|---|
| 入戏 | 4.02 | **4.18** | +0.16 |
| 扣境 | 3.32 | **4.03** | +0.71 |
| **secret_handling** | **2.85** | **4.14** | **+1.28** |
| 泄密翻车率 | **35%(12/34)** | **0%(0/34)** | 12/12 全修 |

**结论**：喂引擎决策+秘密后，**35% 泄密→0%、且入戏微升(+0.16)**——护栏没压平声音、只把它变安全。**「引擎决策，模型配音」架构验证通过：又安全又入戏。** 这直接落到游戏 `chat()` 后端的可实现形态：引擎必须把「哪些秘密不可外露 + 此刻的决策」喂给模型，模型只负责配音。

**⚠️ 但护栏挡不住"生成鲁棒性"翻车——120B 偶尔整段飘日文/夹英文/截断。**

## Phase C · decision-distill 复核 → NPU 决策重构（structural micro-social model）

**🔑 分水岭修正：「teacher=日常现实 / logic=编剧戏剧」多半是 observation 差、不是 policy 差。**
salience_probe：同一批冲突案例，给 teacher 的 packet 补上【心结】（谁冒犯了你·怨气·状态·憋了多久）后，teacher 主动处理冲突 **7% → 88%**，逼近 logic 的 99%；且大量案例 teacher 与 logic **选同一个动作**。之前 Phase-C「teacher 一律去吃饭/睡觉、不理冲突」的结论，主因是**旧 packet 近事为空、confront 候选无来由**——teacher 根本不知道有个疙瘩。补上显著性后 teacher **回归 logic**。**教训**：判「谁更好」前先确保两方看到同样的信息；不然测的是 prompt 不是 policy。（外部 review 一眼点破，和 bake-off 那次 Codex 一样——独立审计 > 自评，第 N 次应验。）

**🔬 Phase-C 四门全崩，但崩得有价值：诊断出三层混淆。**

**💡 这镇的"决定性"只在社交轴，不在需求轴——且冲突都很轻。**

**🧭 重构方向（不是"model decides everything"，是把已验证架构升级一格）：**

## Clean blind judge · 定方向 → CHARACTER 层落地（DSL 首块被独立盲评验证）

**🔑 logic 的"总是当面理论"不是风格、是 out-of-character——盲评 7:1 偏 defer。**
在 clean（P0 修过的）packet 上，对 open-grievance（委屈方）做结构化盲评：**confront（logic 默认）vs defer（让这口气过去=最优非攻击动作）**，10 路并行盲评子代理各只见 {处境,A,B}，confront/logic 身份隐在 mapping 里、镜像翻转消位置偏、A=A 控制校准判平率。结果 **p_eff(confront)=0.12，95%CI [0.07,0.15]**（整段 <0.5）；raw confront 17 : defer 125 : tie 0。**A=A 控制 8/8 判"平"、镜像一致 93%**——评审不是强选噪声，是真判别。说明 logic 死规则 confront 有九成时候"不像这个人此刻会做"。

**💡 最高价值单条规则：把 open-grievance 默认从 confront 翻成 defer——单这条对齐盲评 91%（logic 仅 9%）。**
特征探针：confront 信号几乎全来自**一个人设**——老海(hai)，性格【耿直·好酒】，5/5 判 confront；其余一律 defer（连"莽撞"的小薇也 1:4 偏 defer）。三策略对齐盲评(n=66)：logic(总 confront) **9%** / defer(总让过) **91%** / char(默认 defer + 直性子 confront) **98%**（in-sample、乐观）。**结论**：logic 相对盲评有巨大 in-character headroom，且这 headroom 被一小撮 typed 规则（默认 defer + 人设条件 confront）几乎吃满——不需要大模型在运行时决策。calibrated DSL 现在委屈方 engage 8%（对齐盲评）、冒犯方 100% 道歉。

**🧭 但 in-character ≠ dramatic——两轴分开，CHARACTER 默认 defer / DRAMA 另判是否开场。**
盲评测的是"像不像这个人此刻会做"，不是"对剧情好不好"。一个人设一致的小镇多数时候"让小怨气过去"，但一部生活模拟**可能想要**更多对质来制造戏剧。故 DSL 的 **CHARACTER 层默认 defer**、**DRAMA 层（导演）另行决定**是否为推进 arc 强开一场对质（`W_DRAMA_OVERRIDE`，in-character 模式=0）——两轴**绝不加权混合**（docs/24 的分层正为此）。这条也解释了为何"引擎造机会→模型/规则提响应→theory 裁决"比"logic 决定一切"更对：logic 把 CHARACTER 与 DRAMA 混在一条死规则里，才会九成 out-of-character。

**🛠 盲评前抓到并修的 defer 语义 bug（否则测的是伪对照）。**
"defer" 一度解析到 **rally_oust（撺掇旁人孤立/施压）**——比 confront 更凶，根本不是"让它过去"。只排除 {confront,apologize,mediate} 不够；扩 AGGRESSION 排除集到含 rally_oust/gossip_rep/endorse/leak/gossip 后，defer 才落到真正的维护(去玩耍/吃饭)或示好(找对方聊个中性话题)。**教训**：结构化对照里"另一个选项"的语义必须亲验，否则 A/B 都在攻击、盲评白跑。

**✅ held-out 复验：frozen 人设规则真泛化（98%），不是过拟合。**
把 judge#1 拟合出的规则（默认 defer + confront iff 耿直/好酒/认死理）**冻结**，套到 disjoint 的 judge#2（67 例全新案、seeds 1-8、含 6 例老海）：**char 98% / default-defer 94% / logic 仅 6%**。方向复现 p_eff(confront)=0.08 CI[0.03,0.14]、A=A 控制 8/8 判平、镜像一致 96%。特征探针在新案上**独立重现**同一信号（耿直/好酒 是唯一 confront 关联、persona=老海）。边际发现：sev7+ 0.16 vs sev<7 0.03——高 severity 略偏 confront，可加一个微 severity 项，但 default-defer 已吃 94%。**这条决策的 in-character 政策被"离线 LLM 盲评 + 特征探针 → 蒸馏成 typed 规则"完全吃下，运行时零推理**——是"蒸馏进规则、不是蒸馏进权重"：不需要 NPU/model 在运行时做这个决策，NPU 的价值收敛到 VOICE（动态配音 prefill），决策交给确定性 theory 引擎（logic 6%→char 98% in-character，且可回放、可解释）。

- **agents.json 给 ben/coco 配 `spatial_address` = home/1f** → home_space="home" → 家绑定生效。

**🛑 踩到并修掉的坑：把 energy+fun 都绑在家 → seed1 阿本 fun 归零饿穿（tick1672，人在家里）。**
根因不是"家里没 fun 对象"（桌子有），而是：他 62% 时间在镇上、fun 在镇上【无处可满足】一路衰减；而承诺 pre-empt 只在

## Town-World · #15 观察项收案：搅动，不是稀释（30 个全新 seed 的对照裁决）

**✅ Tier-C 的 #15 「12/12→11/12」被裁决为【轨迹搅动】而非【结构性稀释】——机制一行没改，观察项撤销，加住户的路解锁。**
按 Phase D 的纪律（closed-loop 裁决优先于机制补丁），没有直接上"室内社交 affordance/晚间节律"这类补丁，而是先做对照实验：
**30 个全新 seed（13–42）× 60 天 × (HEAD=Tier-C vs 基线=Tier-C 前一提交的 worktree)**，4 进程并行 ~10 分钟跑完。

结果一边倒：
- **HEAD：#15 失败 0/30**；且 **37 项不变量全绿**（含 #01 无饿穿 30/30）——Tier-C 拿到了迄今最强的 out-of-sample 稳健性证据。
- **基线：#15 失败 2/30**（seed 16: fei(-1.0) 接受率 0.66/镇均 0.50；seed 20）——**基线自己也会在 ~7% 的 seed 上边缘破 #15**。
- 结论：CI seed 1-12 上的 seed-6 反转，只是决策扰动重掷边缘 seed 骰子的正常表现（#15 本底边缘率 ~4-7%），
  与"居民搬进室内"无结构性因果。**Tier-C 在新种子上反而 0 失败。**

**教训（并入方法论）**：
1. **软不变量的 Δ 要先在【全新 seed】上重裁决，再谈机制修补**——12 个 CI seed 的 ±1 反转，在 ~5% 本底边缘率下是常态
  （P(≥1 反转 in 12)≈40%），把它当结构性回归去"修"，反而会像 Phase D 的 BLUNT 扩项那样引入没有因果依据的改动。
2. **加住户的门槛升级为「宽 seed #15 检查」**：以后每次给室内加居民，跑一遍 30-seed × 60d（--det 0，~7 分钟）确认 #15
  失败率不高于本底，比盯着 12-seed 门的单个反转靠谱得多。
3. 对照实验的正确姿势：**基线用 git worktree**（不动工作树）、两变体各拆两块种子段并行、--det 0 省掉重复确定性跑。
