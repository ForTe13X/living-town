## §1 手机端可行性：APK + 骁龙 8 Elite

**结论：可行，不需要重写任何核心系统；最大风险不是性能，是跨平台浮点确定性（ARM libm vs x86 超越函数 ULP 差异可能破字节级 digest）——必须做第一道验证门。**

### 1.1 关键查证（本次实抓）
- **NobodyWho 官方 Android 库已存在**：PR [#201](https://github.com/nobodywho-ooo/nobodywho/pull/201)（2025-08-26 合并，作者在 OnePlus 13=骁龙 8 Elite 实机验证）；`nobodywho-godot-v9.4.0`（2026-06-15）release zip 经 central-directory 解析确认内含 `libnobodywho-godot-aarch64-linux-android-{release,debug}.so`。**零自编译。**（M0 时再 `unzip -l` 复核一次。）
- **骁龙 8 Elite 公开实测 tok/s**（[Qualcomm IWOCL 2025 slides p14](https://www.iwocl.org/wp-content/uploads/iwocl-2025-hongqiang-wang-lamacpp-backend-update.pdf)，llama.cpp OpenCL Adreno 后端，Q4_0）：1.5B decode **43 tok/s**（prefill 384）；Gemma2-2B 21；Llama2-7B 15；Llama3-8B 11。3B 内插 ≈17-21 tok/s（与带宽公式 76.8GB/s×50%÷1.9GB≈20 吻合）。
- **NPU 反直觉**：Genie/QNN 路线 3B≈10、8B≈5 tok/s（[grapeup 实测](https://grapeup.com/blog/running-llms-on-device-with-qualcomm-snapdragon-8-elite)）——decode 同受内存墙，不比 GPU 快，且要脱离 GGUF；仅远期观察（llama.cpp 主线已有 Hexagon 后端，将来可能零成本受益）。
- 注意：官方 android .so 大概率 **CPU-only**（NEON+i8mm）；IWOCL 的 OpenCL 数字是"若自编译开 OpenCL feature"的上行情形。CPU decode 同受同一条 DRAM 带宽约束，量级不变；prefill 折半仍在秒级。

### 1.2 延迟与档位（校验修正后口径）
外推公式 = 780M 实测 × 带宽比(102/76.8≈1.33) × 热态系数(1.2-1.5)：
- **1.5B 决策 p50 ≈1.7s(冷) / 2.1-2.6s(热)** → 探针判 **host 档**（fast<1.5s / host<6s / slow<15s，AIBackend 实名）；对话流式 ≈30-40 汉字/s，气泡体验成立。
- **3B ≈3.9s(冷) / 4.6-5.8s(热)** → host 贴 slow 线，降调用频率用。8B 8-10s 贴 12s 截止线，不建议。
- 退路链完整：SLM(CPU) → llm 后端连局域网 LM Studio（现成零改动）→ logic 地板。

### 1.3 引擎与移动特有问题
- **tick 外推**：容器基线 14.8µs/NPC·tick(LOD) × Oryon 保守系数 1.3-1.7 → 真机 ≈19-25µs。60fps 判定（speed1=12.5tick/s，尖峰须<16.7ms）：全保真 N=120 ✔ / N=240 ✖；LOD cap12 N=240 ✔（6ms）/ N=480 贴线（12ms）/ N=960 需 **tick 分帧切片**（按 tick 内固定顺序切、不依赖帧时间→确定性无损，需 digest 门回归）。渲染（2D 像素 + Adreno 830）<1ms 量级非瓶颈；**瓶颈在且只在 GDScript 单线程 tick 尖峰**。
- **模型分发与许可（铁律④）**：随包只能 **1.5B-Q4（Apache-2.0，1.1GB）** 走 Play install-time asset pack（PAD 额度 4GB；base AAB ≤200MB，本体约 100-150MB 可行）；**3B 是 qwen-research 许可，商用不可随包**——只做"自选下载/导入 GGUF"入口默认关。GGUF 必须放 `user://`（llama.cpp 要真实路径 mmap，进不了 PCK）+ SHA256 校验失败回 logic 地板。
- **热节流**：8 Elite 持续负载稳定度 50-70%；结构性优势=算力探针本来就按 p50 定档，热衰减=p50 上升=自动降档——只需把探针改**滑动窗口持续采样**。规划一律按热态取数（上文已按此口径）。
- **电耗（校验重写）**：per-sim-日口径掩盖真相——speed1 下 1 实时小时≈187 游戏日，串行 LLM 若近饱和（每 2s 一次 15-20J）≈电池 40-50%/h。**电耗上限由"串行 p50×duty cycle"决定**：目标 1h<15%（平均附加≤2.9W）需 duty cycle ≤30-50% → 令牌桶加**实时窗预算**（非只按 sim-日）+低电量→logic 一行策略。
- **内存**：16GB 机型 1.5B 方案应用总占 ≈3GB 安全；3B ≈4.2GB 可跑但后台易被 LMK 杀 → `onTrimMemory` 卸模型、前台重载（1-2s）。
- **生命周期**：pause→确定性快照（tick号+种子+状态，goto_tick 体系天然适配）→取消 in-flight LLM 走兜底（决策=候选下标，丢一次零副作用——架构红利）。
- **触控 UI（唯一必写新代码，纯表现层）**：虚拟摇杆（左半屏）+ 靠近目标 radial menu（M1 玩家动作直接映射）+ 底部抽屉 + `canvas_items` 整数倍像素缩放。
- **Play 政策**：AAB 强制、targetSdk 35、16KB page size（M0 用 zipalign 查 .so 对齐）、数据安全表单（无采集如实报）、AI 生成内容政策要求举报入口——长按气泡举报/关 AI 开关即满足（顺便就是"无模型可玩"开关）。

### 1.4 Roadmap（每步验证门）与角色数量结论
- **M0 导出打通**（1-2 天）：arm64 APK 真机启动。
- **M1 跨平台确定性门（最高优先）**：同 seed 20 天，桌面 vs **arm64 真机**逐日 digest/event_digest 逐字节比对（x86_64 模拟器测不到 ARM libm 差异，只做功能冒烟）。失败→定位超越函数→查表/定点化。此门不过，手机版=另一个宇宙，回放全废。
- **M2 真机 tick bench 标定**：重跑 N∈{60..960} 两组，拟合真机系数替换外推（若差于 2.0× 按公式重算结论表）。门：N=240 LOD 60fps 掉帧<1%，30min 热浸泡后仍达标。
- **M3 SLM 接入**：官方 .so + 1.5B→user://，探针实测。门：p50≤3s(冷)/≤5s(热)，0 次超 12s，兜底率<5%，mock digest 不变。不达标才开 OpenCL 自编译分支。
- **M4 移动化收尾**：触控/生命周期快照/PAD+SHA256/实时窗电量策略。门：杀进程恢复 digest 连续；1h 电耗<15%；AAB 过内测轨。
- **M5 可选**：3B 自选入口（许可告知）、OpenCL/Hexagon、120Hz。

## §2 潜力上界：AI Max 395 及同代 APU 理论天花板

**结论：token 生成带宽受限（tok/s≈有效带宽÷模型字节），395 实测效率 ≈43-52%；"有声镇子"的规模瓶颈在引擎不在 LLM——因为 L5 预算制使 LLM 调用∝预算而非 N，任意 N 下进入玩家注意力半径的 NPC 都可有声。**

两个修正项：①**prefill 是算力墙**非带宽墙——RDNA3.5 对 70B 长 prompt 可能 15-40s/2k token → 70B 只能干**离线异步活**；②KV cache 70B@4k(GQA)≈0.3-0.7GB，96GB 无压力。**许可雷**：Qwen2.5-72B/3B 均非 Apache-2.0；商用 70B 级编剧选 Llama-3.3-70B（社区许可）或 Apache 的 Qwen 天花板 32B/14B。

### 2.2 预算·并发·有声上界
- speed1 每 sim-日=19.2s：3B 短 prompt p50 0.55s → 单 slot 饱和 ≈35 次/日（budget=40 已逼近饱和线，L5 排队行为设计自洽）；**深化 prompt 2.8s → 仅 ≈7 次/日**——解法=prompt 前缀缓存，但**当前是 per-call 一次性 NobodyWho worker（KV 随 worker 销毁）**，须先改持久 worker 或走 llama-server（llm 后端现成）才能把 2.8s 砍回 <1s。这是 LLM 侧第一优化项。
- **有声 NPC 无 N 硬上界**（实测依据：fired 3093→846 封顶）：上界是玩家注意力（阅读 ≈1-2 条/s，3B 的 93字/s 已超过）。正确表述=任意 N 下注意力半径内全有声，其余 logic 地板驱动且行为分布不劣化。
- **多模型编排（96GB UMA 独有）**：3B(决策/反思)+8B(玩家对话/近场有声)+70B(每 sim-周异步编剧) ≈50GB 常驻可行。**确定性接法（关键）**：70B 产出不进 sim，产出 **DataScenarioProvider 纯 JSON 场景**（rift 通路已验证），schema 校验+落盘存档→回放读同一 JSON，digest 不破。带宽共享：70B 解码期间 3B 延迟≈2×→错峰到夜间/暂停，探针加"大模型占用中"感知。

### 2.4 引擎侧上界与下一个瓶颈
- 拟合（容器，三点 N=60/120/240）：全保真 `t≈0.33+0.0017N+2.49e-4N² ms`——**残余 O(N²) 还在**（两两邻近/社交对枚举）；激进 LOD 线性 0.0148ms/agent。
- 上界：@10Hz 全保真 N≈600(容器)/850(真机2×假设)；LOD N≈6750(容器)。**60fps 的真约束是单帧尖峰**：无切片 LOD N≈1100(真机)；**tick 分帧切片**（tick 内固定顺序切、digest 在 tick 完成点算）后纯 GDScript ≈**N=4000-6000 @speed1**（speed8 跌回 600-1000）。当前声明 N≈1000 是稳妥值。
- 下一个瓶颈排序：① O(N) 全量遍历（dormant 也每 tick 摸）→事件驱动唤醒队列（next_wake 最小堆，成本∝活跃数）；② 夜间 O(active²)；③ GDScript Dictionary/Variant 装箱（profile 通常 50%+）；④ 渲染 N>1000 换 MultiMesh；⑤ 信号风暴→批量脏标记。
- **N=5000-10000 路线**：热函数移 C++/GDExtension（Dictionary 重代码 10-40×）或 C#（3-8×）+ SoA 化 + **两阶段确定性多线程**（按 agent_id 固定分片；阶段1 各线程从不可变快照并行算 intent，阶段2 单线程按 id 升序 apply；RNG per-agent 流与调度无关；禁跨线程浮点累加；digest 在 apply 后算=与单线程逐字节等价）。工程量 C# 2-3 周 / C++ 4-6 周，33 不变量+CI 门全程护航。
- **内存唯一定时炸弹=relationships**：每有向对 ≈600B Dictionary；N=1000 全稠密≈0.6GB，**N=5000≈15GB 爆**。治理：top-k=64 确定性剪枝（夜间 familiarity<ε 且 30 日无交互衰减剪除；N=5000→192MB）→SoA 后 9MB。event_log 无界→**digest 链式折叠**（`digest_d=H(digest_{d-1}||events_day_d)`）+JSONL 落盘归档，内存留 7 日窗口；落盘语料顺手喂 70B 编剧。

### 2.5 四个顶配愿景（铁律全守）
- **V1 全声剧场镇（395，≈1 周）**：N=350-500 全保真（**N=500 需把 tick 切片提前进来**——真机单 tick≈35ms 超帧预算；speed 上限≈2×）+ 三模型常驻 + 70B 周更编剧走 JSON 场景注入。
- **V2 千人镇（395，≈2 周）**：N=2000 激进 LOD + 切片 + MultiMesh + relationships 剪枝。
- **V3 万人史诗镇（4-6 周工程投入）**：C++/多线程/唤醒队列/SoA/log 折叠全齐 → N=8000-10000。这是本架构的真极限，再往上超出消费级命题。
- **V4 口袋镇（8840U/手机档，已基本验证）**：N=240-300 LOD+1.5B。**同一存档、同一 digest——同一个镇子在全档位字节级同回放，本身就是卖点。**

**首要标定实验（2026-07-02 已做②③，结果回写）**：
- ② **容器→真机倍率实测 ≈0.8-1.7×（模式相关），非假设的 2×**：native 395 全保真 N=120=16.1s / N=240=45.8s（容器 19.8/77.8 → 1.2-1.7×）；激进 N=240=15.7s / N=960=83.9s（容器 13.3/68 → 0.8-1.0×，真机反略慢）。**结论：容器数字可直接当真机数字用，本节所有"真机 2×"档位下修为容器口径**（全保真@10Hz N≈600；LOD 60fps 无切片 N≈1100；切片后 ≈4000-6000 维持——该档由均摊主导受倍率影响小）。
- ③ **"深化 prompt 2.8s"是冷启动假象**：GPU 实测 per-call worker 第 1 次 2.93s（含 warmup）、第 2 次 671ms；持久 worker 三连问 563-676ms **无额外优势**（GPU prefill 本来就快）。生产管线算力探针启动即打暖 → **热态深化决策 p50≈0.6-0.7s，无需改 AIBackend**（持久 worker/llama-server 仅在 CPU/手机档 prefill 弱时再评估）。每 sim-日饱和 ≈28-30 次（非 7 次）。
- ① **70B smoke 已做（2026-07-03，Llama-3.3-70B-Q4 @LM Studio/395）**：热态 decode **4.72 tok/s**——高于公式带 2.6-3.3、命中社区锚点 4-5（**大模型带宽效率 ≈74%，远高于小模型的 43-52%**——权重读取摊薄了 per-token 开销，效率随模型增大而升，§2.1 表中 32B/70B 档按此上修）；**prefill ≈82 tok/s**（3441 token 实测 44.4s，落在预测带 50-150 内）→ 2k 导演 prompt ≈25s；冷载 ≈95s（40GB，保持常驻或接受按需加载）；长上下文注意力正常（3.4k token 记录中数对 7 位居民）。**结论：每 sim-周 500-1000 token 剧本 ≈2-4 分钟，夜间异步编剧完全可行**，走 DataScenarioProvider JSON 注入保 digest。

## §3 世界系统扩展：roadmap + modular update 方案

**结论：先补挂点缺口（最大真缺口=效果提交挂点 ActionExecutor——CandidateProvider 加的新动作现在会被误套 greet 通用效果污染事件流），再按"物质地基→时间戏剧→社会纵深"三波推进；每个系统=标准落地包五件套，off 状态逐字节不变是每次合入的铁门。**

### 3.2 候选系统（机制核心 + 分类）
1. **经济【原语 Ledger】**：coin 进 inventory（键任意零结构改动）；商店=advertise 加 `{price,stock}`；物价=每夜确定规则带 clamp 波动带。
2. **职业作息【P；←经济】**：job=工位对象 advertise 加 `{job_id,wage,shift_phase}`——与 rhythm.json 天然契合；缺勤=复用爽约→resentment/standing/gossip，**职场戏剧免费**。
3. **生命周期【P】**：age 每夜+1；birthday=hash(id)%365；搬入=克隆路径 append 尾部；**搬出绝不删 agent**（数组序=RNG 流=digest 的根）→ `moved_out` 标志在 `_nightly_active_ids` 等单点过滤。
4. **健康/情绪【数据为主】**：health=needs.json 加一条+诊所对象——**纯数据即通**；低 health→SURVIVAL_GATE 免费获得"卧床优先"；照护=复用 S3b aid。情绪=账本 delta 确定函数纯显示（docs/10 C4：不做独立情绪栈）。
5. **季节/天气【原语 WorldStateProvider】**：weather(day)=`_hash01(seed+day)` 纯哈希查表——**不消耗 RNG 流不存状态**；三张 dampen-only 乘子表（rhythm 1.8× 全镇同步崩溃是实测教训，照抄 rhythm 门控）。
6. **节日/集体事件【P+WorldPatch】**：调度=确定函数（day%7 集市、晴∧day%30 灯会）；Director v1=规则体撒"机会地形"（临时高 advertise 对象→人群自发聚拢）+mass invite 复用承诺全套→集会 discuss/gossip 密度自然升=意见/派系加速器。
7. **治理/选举【P；前置 TOPICS 数据化（Sim.gd 共 6 处引用，值=默认逐字节门）】**：**S2 attitudes 就是选票**（|a|<0.15 弃权=FACTION_SALIENT 现成）；计票=快照纯函数=硬不变量；竞选期=utility 一个键+FJ 动力学天然民意；结果=WorldPatch/调参键。**LLM 演讲（2.8s 级已实测）纯渲染，一票不改。**
8. **住房/产权【P；←经济+生命周期】**：低优先，等贫富分化成形。
9. **技能成长【数据+小 P】**：计数器同构 complementSeen；单调递增零 RNG。
10. **多镇【SKIP→远期】**：廉价替身="信使 NPC"带外镇 gossip（belief 注入，纯数据场景拿到 90% 叙事价值）。

### 3.3 三波 roadmap 与解锁链
- **Wave 1 地基**：1a 挂点补全（S0 逐字节+det+replay_drift==0 三门，零新行为）→ 1b 经济最小闭环（吃饭收费+做活发薪，金钱守恒进 HARD_IDS）∥ 1c 天气（每步单独过门，绝不一次引入多变量）。
- **Wave 2 时间与戏剧**：2a 职业（工资可重算=硬）→ 2b 节日（spawn/despawn 配对=硬；灯会场景两跑 byte 一致）→ 2c 技能。
- **Wave 3 社会纵深**：3a 选举（**S2 意见+S3 派系+L5 预算三套已落地系统的收获期**，第一次共同产出"镇级剧情"）→ 3b 生命周期 → 3c 住房+健康。
- 解锁链：经济→职业→贫富分化→阶层 gossip（gossip_rep subject 从人品扩到财富传闻，S1 管线原样复用）→住房；天气→节律调制→节日→集会密度→选举民意。
- LLM 接入点全部=渲染/提议层吃 L5 令牌桶：物价抱怨=context 注入经济摘要；演讲=长发言；节日祝辞=意图菜单 pick；拔模型全退化罐头照绿。

## §4 三线交叉的共同结论
1. **架构的护城河被三线独立确认**：logic 地板+LLM 只 pick 下标+L5 预算制，使 ①手机上丢一次调用零副作用、②有声 NPC 无 N 硬上界、③新系统 LLM 接入全在渲染层——同一个设计决定在三个方向同时兑现。
2. **确定性是贯穿三线的第一验证门**：手机=跨平台 libm 差异（M1 门）；上界=多线程/切片的等价性（digest 门）；扩展=off 状态逐字节不变（合入铁门）。
3. **"同一个镇子全档位字节级同回放"**（手机 N=240 ↔ 桌面 N=500 ↔ 万人镇同一存档格式）是产品级卖点，值得写进宣传语。
4. 近期最高杠杆动作（跨线共识）：①395 三个标定实验（70B smoke/真机倍率/持久 worker+cache_prompt）；②Wave 1a 挂点补全；③手机 M0+M1（确定性门）。
