# 14 · 可扩展性与规模 — 设计与实测

> 回应三问：①场景/行为策略空间可扩展化（松耦合 / plug-and-play / modular / configurable / 优雅 inject&update）；②模拟时间拉长 + 行动↔时间同频；③更大群体（上界估算 + scaling 方向）。
> 方法：三线并行深研 + 逐线对抗式校验（确定性安全）→ 综合。**五铁律全程守**：确定性+字节级可复现（digest/goto_tick/event_digest）、无模型可玩（logic 地板）、消费级硬件、无版权、复用优先。
> 状态图例：✅ 已实现并验证 · 📐 已设计+对抗校验（未落地）· 🔬 已实测诊断。

## §1 · 行为/场景策略空间的可扩展化 ✅（骨架+数据驱动场景已落地，2026-07-01）

> 已实现：`scripts/SimExtensions.gd` 注册中枢 + `Sim.ext` 字段 + 三挂点（scenario/candidates/nightly，`ext=null` 短路 → **S0 12/12+det 3/3 逐字节不变**）+ `scripts/DataScenarioProvider.gd`（纯 JSON 场景）+ **UtilityTable**（`data/utility.json`+`Sim._w()`：接受阈值/打分权重数据化，改 JSON 即调平衡）。**实测**：① `--scenario rift` 经注册中枢加载 `data/scenarios/rift.json`（零 GDScript 加场景）→ 派系=2 成形、endorse/rally_oust 涌现、33/33 全绿、两跑 byte 一致；② utility.json 值=默认 → S0 逐字节不变，改 `accept_greet=200` → greet 接受骤降到 0（knob 确 live）。守住两坑：注册只在注入时、`freeze()` 幂等。剩余（低优先，可随内容需要再做）：AcceptanceModifier 挂点 + NightlyHook/CandidateProvider 迁移（步骤2/5/6）。

### 现状与耦合点
扩展当前只能改核心大函数（全在 `Sim.gd`）：
- **场景**：`scenario:String` 是裸字符串，`_seed_scenario()` 用 `if/elif betray/faction/freerider` 直接改 agents/beliefs/pacts；新增场景=改此函数。
- **候选（动作维）**：`agent_candidates` 硬串 `_object_candidates`+`_social_candidates`（~100 行巨函数，13+ 动作类型/门槛/效用公式全内联）+`_attend_candidates`。
- **接受（评估维）**：`_acceptance_rule` 是按 action 的 if/elif 阈值链，`_faction_term` 等修正内联。
- **夜间（演化维）**：`_nightly` 顺序写死且顺序敏感。
- **效用权重**：散落魔法数字面量（如 greet `urgency*0.7+aff*0.1+..+6.0`）。
- 同一动作在**候选/接受/执行三处**分别硬编码 → 加一个动作要三地同步改。

### 设计：SimExtensions 注册中枢（复用 `backend=null` 鸭子类型范式）
`Sim` 持有 `var ext: Object = null`（与 `var backend` 同构：headless 可注入、null 走旧路径兜底）。四类可选契约（`has_method` 探测）：
1. **ScenarioProvider** `id()/seed(S)/distorts_harmony()` — 取代 `_seed_scenario` 的 if/elif 与 Invariants 的字符串比较。
2. **CandidateProvider** `id()/candidates(S,ag)->Array` — 追加候选。
3. **AcceptanceModifier** `id()/modify(S,actor,target,action,subject)->float` — 返回叠加到 `_acceptance_rule` 决策和的增量项。
4. **NightlyHook** `id()/order()->int/run(S)` — 按 `(order,id)` 确定序跑每日机制。
外加 **UtilityTable**（`data/utility.json`）：候选构造用 `ext.weight(name, 今日字面量)` 查表，缺省回落。
数据驱动场景：`DataScenarioProvider` 读 `data/scenarios/<id>.json` 声明式 patch，零 GDScript 加简单场景。

`Sim` 侧改动极小、全部 null 短路：`_seed_scenario`/`agent_candidates`/`_acceptance_rule`/`_nightly` 末尾各加一行 `if ext != null: ...`。内建化：现有 betray/faction/freerider 搬成 ScenarioProvider（逻辑逐字节不变）。

### 确定性（校验确认 safe，但有两个**必须先钉死**的坑）
- **无 live RNG / 无 Time / 无字典插入序依赖**：provider 抖动一律经 `_rng_at(salt,who)` 子流；provider 数组在 `freeze()` 里按 `id()`/`(order,id)` `sort_custom` 排定，tick 内只按只读数组遍历。
- ⚠️ **坑1（回放杀手）·注册/freeze 幂等性**：`goto_tick` 反复调 `start_new`。若内建 provider 在 `start_new` 内注册（freeze 前），每次重演都 `append` → 数组翻倍 → digest 漂移。**修正**：注册只在 `SimScript.new()` 后由调用方做一次（与 `backend` 注入同位，`start_new` 不碰注册）；或 `freeze()` 开头先 `clear` 再从注册源重建。
- ⚠️ **坑2·AcceptanceModifier 等值**：内建 modifier 必须逐动作**等值复刻** `_faction_term`——对原本不叠 fac 的动作（confide/leak/aid）返回 0，否则翻转低阈值动作的接受率、扰动软不变量 #26。
- **纠错**：Invariants 的 `harmony = (scenario=="")` 使**任意非空场景字符串被无差别豁免**（过度豁免风险），而非"新场景被误判和睦"。`distorts_harmony()` 的价值是把"非空即豁免"升级为"场景显式声明是否扭曲"。

### Roadmap（每步一可 ship 增量 + 现成 bench 验证；先追加式后替换式）
1. ✅ 骨架 + 三挂点（scenario/candidates/nightly）`ext=null` 短路 → S0 12/12+det 3/3 逐字节不变。
2. NightlyHook（最独立，注册放 `new()` 后一次）→ Harness det + LodAblation 三门。
3. ScenarioProvider（内建三场景搬迁 + harmony 改查 `distorts_harmony`）→ 三场景 `export_trace` 逐值 diff。
4. ✅ UtilityTable（`data/utility.json`+`Sim._w(key,default)`；已抽 9 接受阈值 + obj_dist_penalty/greet_base）→ S0 12/12 逐字节不变（值=默认）；实测 `accept_greet=200` → greet 接受骤降到 0（knob 确 live）。剩余打分魔数按需续抽。
5. CandidateProvider（**仅追加式新动作、排末尾**，已有族顺序零改动）→ `record_decisions`→`set_replay` 核对 `replay_drift==0`。
6. AcceptanceModifier 挂点 + 抽 `_faction_term`（对不叠 fac 的动作返 0，等值）→ 软不变量 #26 不漂。
7. ✅ DataScenarioProvider（纯 JSON 声明场景 `data/scenarios/*.json`）→ `--scenario rift` 派系=2、33/33、两跑 byte 一致。

## §2 · 行动↔时间同频 ✅（已实现并验证，2026-07-01）

### 长跑记账：经济体 commensurate、时钟零漂移（🔬 实测）
一天=240 tick（≈19.2 实时秒@speed1）。decay×一天：hunger 67.2 / energy 43.2 / social 36 / fun 38.4 / hygiene 26.4（皆<100）。最优动作净恢复为正（eat +55/18t、sleep +62/40t…）。维持全需求所需 use 时长 ≈ **89.4t/天 = 37.3%**，留足赶路+社交余量 → 无结构性单调恶化/恒饱和。**sim 时钟是纯整数 tick，`240|TICKS_PER_DAY`、day 边界精确、唯一浮点 `_accum` 只在 `_process` 墙钟层不进状态** → 长跑字节级可复现。
`sim_soak --trace-rhythm` 实测轨迹：need 均值第 5 天起收敛到 81–90 稳态带、30 天不漂 → 佐证。

### 现状缺口 → 同频设计
`tod` 从不进任何确定性决策路径（`_logic_decide`/candidates/`_acceptance_rule` 都不读），只喂 LLM prompt + 画面色调 → **昼夜是纯滤镜、对行为零意义**。
**方案**：`data/rhythm.json` 时段偏好乘子（纯数据、`tod=f(tick_no)` 纯查表、可回放、缺文件即退化）。`_phase_of/_phase_pref`（`Sim.gd`）。接入点：`_object_candidates` 的**收益项**（不动距离惩罚）乘 `_phase_pref(need, tod)`。
**关键守卫**（守 HARD#1 无饿穿）：① 仅当 `_min_need(ag) >= SURVIVAL_GATE`（无紧急需求）时启用节律，任一需求告急 → 节律关闭 → 纯 urgency 主导、绝不因时段延误进食；② **dampen-only**（乘子 ≤ 1.0）：偏好只让 agent"推迟"非当令需求、绝不比基线更渴求 → 物件吸引力不膨胀、不挤占社交（这修掉了强节律=1.8× 时把全镇同步到同时段忙碌、导致 invite/meet 归零 + 派系失衡的回归）；③ 引擎侧 `clamp(0.3,2.0)`。

### 验证（全绿）
- `sim_soak --trace-rhythm` 30 天：**同频信号显现**——energy(睡) 集中暮/late（38%/50%）vs 昼 13%；hunger(餐) 峰在晨(33%)/昼/暮；fun 峰在昼(28%~35%)。33/33 不变量过、无饿穿。
- Harness 12 seed × 60 天：**12/12 全绿 + 确定性双摘要 3/3**。
- LodAblation N=80：保守门+激进门**双 PASS**（节律是确定函数，不破 LOD 等价）。
> 调参零改码：`data/rhythm.json` 的 phases/prefs 即可。此设计与 §1 注册表天然对齐（未来每策略自带 rhythm_key）。

## §3 · 更大群体 — 上界估算与 scaling 方向 🔬（实测诊断）

### 实测（sim_soak，20 天，full-sim vs 激进 LOD）
| 配置 | N=60 | N=120 | N=240 | N=480 |
|---|---|---|---|---|
| full-sim（LOD off）墙钟 | 4.7s | 18.1s | 76.1s | — |
| 激进 LOD cap=12 墙钟 | — | — | 76.2s | 406.8s |
（减启动常量 ~1.7s）。**两条曲线都 ~O(N²)：N 翻倍 → 时间约 ×4~5。**

### 诊断：真瓶颈不是候选枚举，而是**未 LOD 化的夜间 O(N²)**
- 激进 LOD 把候选枚举 `cand_calls` 砍 83%（∝ cap 而非 N，已由 LodAblation 证），**但墙钟与 full-sim 几乎相同** → 候选枚举不是瓶颈。
- 真凶：`_form_pacts_greedy`（`for id in ids: for oid in ids:` 嵌套 + 每对 ~6 次 `_rel` 字典查 = **O(N²)**）与 `_recompute_factions`（O(N×medoids)），**每夜为全体 N 跑、与 LOD 无关**。N=480 的 20 夜 × 480² ≈ 4.6M 对 × 常数 → 主导墙钟。
- 次要（长跑相关）：`_resolve_commitments`/`_sweep_conflicts`/`_has_active_meet` 每 tick 扫**累积增长**的列表 → O(累积) ∝ N·time。**已修 commitments**（✅ 见下），conflicts 类同、体量小、待办。

### 已落地优化 ✅：commitment 活跃工作集（服务"模拟时间拉长"）
`_active_commitments` 只含未决承诺（创建时 append、解算时剔除）；`_resolve_commitments`/`_has_active_meet`/`_any_active_meet` 只扫它 → per-tick 成本 ∝ **未决数(∝N)** 而非**累积总量(∝N·time)**。`commitments` 全量保留供不变量。**语义等价**（同序处理活跃子集）→ S0 12/12 + det 3/3 逐字节验证，承诺不变量 #08/#09/#10 全绿。

### 上界估算（消费级机器，logic 地板，headless）
- **full-sim**：夜间 O(N²) 主导 → 实用上界 **~N=150–200**。
- **激进 LOD + R1 + R2（均已实现）**：夜间 O(cap²) + 社交嵌套 O(N²) 消除 → **标度回落到 ≈线性**（~×2.2/翻倍）。实测 tick-loop 20 天：N=240=13.3s / N=480=28.3s / **N=960=67.6s（≈3.4s/sim-日）**。**当前实用上界 ~N=1000**（headless soak；实际游戏不每帧跑 check_all，gameplay 成本更低）。
- 内存：event_log ∝ N·time（需分段滚动，见 R3）；agent 状态 ∝ N 平凡。

### Scaling roadmap（按杠杆排序；每步用 LodAblation/cand_calls/墙钟客观度量）
- **R1 ✅ 已实现·夜间 LOD 化**：新增 `_nightly_active_ids()`——激进 LOD(`lod_aggregate`+`lod_near_cap>0`) 下只返回 near 集(定序)、否则全体；`_form_pacts_greedy`/`_recompute_factions` 均改用它。far agent → factionless（attitudes 冻结、行为上不引用派系、near 不与 far 结盟 → 语义等价、便宜）。全量配置(全/保守)逐字节不变。**实测墙钟（20 天）：N=240 76→32s(2.4×)、N=480 407→104s(3.9×)**；标度从 ~×5.3/翻倍降到 ~×3.2。**S0 12/12+det 3/3 逐字节不变、激进门(LodAblation N=80) 硬不变量全绿。** 残余超线性来自密度驱动的 per-tick 社交（near agent 在拥挤地图里 `_nearby_agents` 仍 O(N)）→ R2 territory。
- **R2 ✅ 已实现·消除社交嵌套 O(N²)**：profile 定位到真凶不是 `_nearby_agents`，而是 `_social_candidates` 里 gossip_rep/endorse 两处 `for c2 in agents`（**在 `for o in _nearby_agents` 内层** → O(nearby×N)=O(N²)/agent）。修法：在邻居循环前**预算 `bad_targets`**（我对其有坏名声 ≤REP_GOSSIP_TH 的第三方，按 agent 定序、仅查已存在关系不 auto-create），两处内层改遍历它（通常极小）→ O(N²)→O(N)。**语义等价**：未建关系者 standing=0>阈从不入选，故 `bad_targets` 与原扫全量同集、同序、同首个匹配 → 事件逐字节不变（digest 基于 event_log，零名声关系不产事件）。**实测 tick-loop（激进 cap=12,20天）：N=240 30.7→13.3s、N=480 104.8→28.3s、N=960 616.7→67.6s(9.1×)；标度从 ~×5.9/翻倍降到 ~×2.2（≈线性）。S0 12/12+det 3/3 逐字节不变、激进门 PASS。** 残余线性项：`bad_targets` 的 O(N) 预算 + near-set 排序 O(N log N)（R4）。真正的 `_nearby_agents` 倒排桶留待更大 N（当前 O(N) 线性、非瓶颈）。
- **R3·event_log 分段滚动**：长跑内存 ∝ N·time → 滚动窗口 + 段摘要（event_digest 已是 O(1) 折叠，天然兼容）。
- **R4·near-set 增量维护**：`_compute_near_set` 现每 tick `duplicate+sort_custom` O(N log N)；agent 每 tick 只挪 1 格 → 改增量堆/脏标记维护，均摊 O(log N) 或 O(1)。
- **R5·离屏群体标量聚合（S-scale-5）**：把远端"背景人群"从逐 agent 降为区域标量密度场（人数/情绪聚合），只在靠近相机时实例化（配合 L6 palette-swap 即时派生外观）。冲上千的终极手段。
- **conflicts 工作集**：同 commitments，把 `_sweep_conflicts`/`_find_conflict` 切到活跃集（体量小、优先级低）。
