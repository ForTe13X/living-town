# 02 · 技术架构 · 混合仿真（确定性引擎为底，LLM 为声音）

> 一句话：**The Sims 的需求/效用引擎当肌肉（确定性、廉价、每 tick 跑），
> generative-agents 的记忆/反思/规划当大脑（LLM、稀疏、异步），中间用「合法候选 + 引擎兜底」的契约缝合。**
> 这既是 22nd 已验证的纪律，也是 2025 学界 generative-agents 的共识。

## 0. 技术选型（结论）

**Godot 4 + GDScript**，复用 22nd 的全套流水线与本地 LLM 集成。理由（详见研究，摘要）：
- 工作室已把**最难的一块——Godot 上的本地 LLM 集成**调通且实测（doc 13），换引擎=重付这笔「LLM 集成税」。
- Godot 一次给 **Web + 桌面** 双端；Web/TS 跑不动本地大模型（浏览器首载灾难），最终还是要 Electron→等于丢掉打包优势。
- Bevy/Rust 只在「上千 Agent」一个轴上赢，而本作 Stardew/Sims 规模（几十~低百 Agent）GDScript 足够；留作 **Dwarf-Fortress 级逃生舱**。
- 像素工具链：Aseprite → **Aseprite Wizard** → `TileMapLayer`(4.3+)，自动重导入。
- 热循环逃生舱：若某条 sim 循环（大网格寻路/大规模需求 tick）变热，**把那一条循环**移到 C#（同工程内，紧数值循环可快 ~100×），不重写全局。

## 2. 确定性需求/效用引擎（肌肉，零 LLM）

**这是底座，必须先做、先好玩、可独立运行。** 取自 The Sims / RimWorld：

- **需求（motives）**：每个 Agent 一组标量需求（饥饿/精力/社交/趣味/卫生…），每 tick 衰减。
- **效用 AI（utility AI）**：世界对象「广告」交互（床→精力，灶→饥饿，广场→社交）。Agent 用打分函数（需求紧迫度 × 距离 × 可用性）从 **`agent_candidates()`** 里选最优 → `agent_apply()`。
- **option-action 分层**（Lyfe Agents）：选中的不是一个原子动作，而是一个会持续一阵的「**option**」（如「去咖啡馆社交」）；option 内部的低层动作（寻路、坐下、喝）由**廉价脚本**每 tick 执行，**不调 LLM**。LLM 只偶发地参与「选哪个 option」与「说什么」。

→ **关键性质**：没有任何模型时，这层让整座小镇照常生活（progressive enhancement 的「地板」）。

## 3. tick / step 调度（引擎核心，取自 AI Town）

- **tick**：~60Hz 在内存里跑物理/寻路/需求衰减/option 推进——**不落盘、不调 LLM**。
- **step**：把约 1 秒的 tick 批量持久化一次（存档/回放点），不要每帧写盘。
- **LLM 调用与 tick 解耦**：所有模型调用是**异步「内部动作」**，其结果不直接改状态，而是作为**游戏输入回灌**（单一序列化点 → 可回放）。NPC 在等模型（1–2s）时继续走它的脚本 option，**永不卡帧**。每个 Agent 一个 `thinking` 标志，防止重复排队。

## 4. PIANO 式并发模块（多 Agent 实时一致性，取自 Project Sid）

**不要每 tick 跑一条庞大的串行 LLM 链。** 把 Agent 的「思考」拆成不同节奏、共享一个 agent-state blob 的并发模块：
- **快反射循环**（每 tick，脚本/极廉价）：需求紧迫处理、避障、即时反应。
- **慢决策循环**（每 N 秒/分钟，LLM）：选 option、社交意图、对话。
- **反思任务**（每日/阈值触发，LLM）：把记忆提炼成洞察。
- **认知控制器（CC）瓶颈**：一个小步骤看「过滤后的状态切片」，定下高层动作并**广播**，让「说的」和「做的」一致（解决「嘴上一套手上一套」的不一致）。本作里 CC ≈ `Director` + 每 Agent 的 option 选择。

## 5. 世界状态树 + 受约束动作词表（grounding，取自 Stanford + Mantella）

- **世界是一棵树**：world → area → object，每个 object 带状态字符串（"灶: 关"）。Agent 只「知道」它观察过的子图，记忆里存「上次见到的状态」——既真实又把 prompt 压小。
- **动作词表是受约束的枚举**：LLM 的输出 `action` 必须落在引擎认可的合法动词集合（`move_to|talk|use|give|idle|...`）+ 合法 `target`（来自世界树）。**LLM 永远不能凭空捏造不可能的动作**——这正是 22nd 的 `ai_candidates → ai_apply` 契约在生活模拟里的推广。

## 6. 确定性与 bench 红线（继承 22nd）

- 所有随机走 `_rng_at(tick, salt)`，种子 = `seed_base + tick*K + salt`。
- **scenario / 回放 / soak-sim** 永远走**确定性后端**（needs/utility AI，固定种子），**不调 LLM**——保证可复现、可自动评测。
- LLM/对话/玩家介入/元进度只在 `interactive`（`scenario==""`）下触发。
- 这样既能要「涌现」，又能在开发期用 22nd 的 Docker judge + 自走子做**可复现的自动评测与平衡**。

## 7. 模块清单（映射到 game/scripts）

| 模块 | 文件 | 职责 |
|---|---|---|
| 屏幕管理/入口 | `Main.gd` | CLI 解析 `--scenario/--days`，切 Title/World/Inspect 屏 |
| **仿真引擎** | `Sim.gd`（autoload） | 世界树 + Agents + tick/step + 需求/效用 + `agent_candidates/apply` + 信号 |
| 记忆流 | `Memory.gd` | append-only 记忆 + recency/importance/relevance 检索 + 反思 + summarize-and-forget |
| 导演 | `Director.gd` | 偶发设定镇上意图（黑板）、点名某 NPC 接近玩家、镇上情绪 |
| LLM 后端 | `AIBackend.gd`（autoload） | `logic\|llm\|slm` 三档，异步、语法约束 JSON、引擎兜底（复用 22nd） |
| 主题/资产 | `Art.gd`（autoload） | 调色板 + 像素资产三级回退 `pro/png>png>svg` + manifest |
| 渲染 | `WorldView.gd` | 像素地图 + Agent 精灵 + 气泡/FX，纯订阅 Sim 信号 |
| soak 仿真 | `sim_soak.gd`（`extends SceneTree`） | headless 跑 N 天，验证需求/社交不崩、采集统计 |

> 各档 LLM 的接线、结构化输出、成本与降级，见 [03 LLM 集成架构](03-LLM集成架构.md)。
