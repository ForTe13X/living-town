# 00 · 现有项目与研发流水线分析（《小鱼岛》@ `June/22nd`）

> 目的：在启动新项目前，先**吃透现有的研发流水线**，因为新项目的最大杠杆就是「复用而非重建」。
> 结论先行：`22nd` 不是一个普通游戏仓库，而是一台**自评估的 AI 游戏工厂**——
> 规格化任务 → 强编码 Agent 生成 Godot 工程 → Docker 无头回放 + 多模态评审打分 → 平衡自走子调参 → 打包发布，
> 外加一套**已实测跑通的本地 LLM 集成**（这正是新项目要继承的核心资产）。

---

## 1. 这是个什么项目

**《小鱼岛 / Little Fish Island》**：从一款中文实体桌游数字化而来的 **Godot 4.6.2** 渔业经营 / **引擎构筑（engine-building）** 卡牌游戏。
单人对一个 AI 对手，10–15 分钟一局，确定性可回放，治愈系动物海洋画风。游戏本体由 **Claude/Opus 直接编写**，
经 `gamecraft-bench` 的格式与评分基线规范，由 Docker 流水线自动评测。GPT-5.5 校准公允分 **0.9525**（基准最强配置 0.41）。

仓库三大件 + 交付：
```
22nd/
  docs/            15 篇交付文档（产品/技术/AI 论证/路线图/平衡/美术/手册/复刻方案…）
  gamecraft-bench/ 评测基准（140 个游戏任务 / 15 个品类 / 评分 harness）
  pipeline/        Docker 评测 + HTML5 打包 + 静态服 + 自走子
  release/         三档发布物（v2 多平台 / faithful 1:1 / 内置AI）
```

> ⚠️ 注意：`June/` 下的**每个日期文件夹是不同项目**（8th=后端/pytest，10th=SPI，24th=Pet，22nd=本游戏，26th=本新项目），
> 不是同一项目的快照。新项目落在 26th，但复用 22nd 的流水线。

## 2. 研发流水线全景（端到端）

```
 ① 规格            ② 生成              ③ 评测/评审            ④ 平衡            ⑤ 打包         ⑥ 发布
 bench task   →   编码Agent写    →   Docker 无头回放    →   自走子300局   →  HTML5/wasm  →  多平台
 instruction      Godot工程        Xvfb+ffmpeg+xdotool     调一个旋钮       dist/ + exe     v2/faithful/
 rubric.json      demo_outputs     多模态judge打分→reward   再跑          serve.js托管    builtinAI
 task.toml        *.json traces    reward.txt/breakdown
```

### ① 规格：一个 bench「任务」是自包含目录
`gamecraft-bench/tasks/<name>/`：
- `instruction.md`——交给生成 Agent 的提示（要做成什么游戏）
- `task.toml`——Harbor 资源/超时（agent 7200s、verifier 1800s、cpu1/mem2GB、`allow_internet=true`）
- `tests/rubric.json`——**隐藏的评分契约**：`build_check` + 分类 `requirements` + 算术 `score_formula`
- `tests/test.sh`——verifier 入口
- `solution/solve.sh` + `solution/files/`——oracle 参考解（`nop` agent 必须得 0）

### ② 生成：强编码 Agent 写 Godot 工程
Agent 把游戏写进 `/workspace/game/`（一个 Godot 工程）+ `demo_outputs/*.json`（确定性输入轨迹）。
`gamecraft_bench/local_agents.py` 驱动 Claude Code / Codex / Kimi 等无头跑：
`claude --output-format=stream-json --permission-mode=bypassPermissions --disallowedTools EnterPlanMode,... --print`，
5xx/429 用 `--resume` 重试。**排行榜**：Claude Code+Opus-4.7 high = 41.46%（最佳），Codex+GPT-5.5 = 39.49，Kimi = 30.65。

### ③ 评测/评审：判「行为」不判「源码」
`pipeline` 是一个 **常驻 Docker runner**（`sleep infinity` + 每次 `docker compose exec`，保持 Godot/Xvfb 栈温热）。
`gc.ps1`（PowerShell）是宿主编排器。两条子流水线：
- **VERIFY**：`verify.sh` 把工程 stage 到 `/workspace/game` → `python -m gamecraft_bench.verifier`：
  1. **build gate**：`godot --headless --import && --quit-after 5`（不过则整分 ×0）
  2. 对每条 trace：`replay.py` 用 **Xvfb + opengl3 真渲染 + xdotool 合成输入 + ffmpeg x11grab 录 mp4**（注意：headless 渲染器截不了图，必须 Xvfb 真渲染）
  3. ffmpeg 采样 PNG 帧（≤40 帧）
  4. **多模态 judge**：一次批量调用按 rubric 给每条 requirement 打分；后端 `stub|openai|claude|gemini|kimi`，默认 **LM Studio**（`qwen3-vl-4b`，OpenAI 兼容，经 `host.docker.internal:1234`）
  5. 跨 demo 聚合（机制取 max、视觉取 mean）→ AST 安全求值 `score_formula` → `reward ∈ [0,1]`
- **PACKAGE**：`package.sh` 拉一次 ~1.25GB Web 导出模板 → `godot --headless --export-release "Web" dist/index.html` → `serve.js` 静态托管（零依赖、处理 wasm/pck MIME + HTTP Range）。

产物落 `out/verifier/`：`reward.txt`、`breakdown.json`、`judge_log.json`、`demos/<id>/<id>.mp4` + `frames/`。
**三档 judge**：stub=1.000（链路自检）、本地 qwen3-vl-4b=0.75–0.96（噪声大，仅 smoke）、GPT-5.5=0.9525（权威）。

### ④ 平衡：headless 自走子
`gc.ps1 sim [N]` → `godot --headless --script res://sim.gd -- --games N`（默认 300）。
`sim.gd extends SceneTree`，**只驱动规则引擎、无 UI/网络**，贪心玩家 vs 启发式 AI 跑 300 个确定性种子，报胜率/回合统计。
纪律：**一次只调一个旋钮 → 重跑 → 看健康带（46–53% 胜率）**。balance 全在 `balance.json`。

### ⑤⑥ 打包发布：三档产品线
- `release/xiaoyudao-v2/`——多平台「轻量版」（win exe / linux / web wasm + serve.js / promo / docs），~124MB
- `release/xiaoyudao-faithful-win/XiaoyudaoFaithful.exe`——**1:1 忠实复刻**官方 v4 规则（独立 `Faithful.gd` 引擎，七个计分通道，二选一回合）
- `release/xiaoyudao-builtinAI-win-test/`——`LittleFishIsland.exe` + `libgodot_llm...dll` + `models/qwen2.5-0.5b-instruct-q4_k_m.gguf`（491MB）：**完全离线**的本地 SLM 对手

## 3. 代码架构（Godot 工程，值得新项目照抄的范式）

**Autoload 单例 + headless 规则引擎 + 程序化 UI** 三段式：
- 3 个 autoload：`Art`（主题/资产解析）、`Game`（规则+状态）、`Codex`（持久化）；`Main.gd`（Node2D）做屏幕管理器，CLI 解析 `--scenario` 并交叉淡入 Control 屏。
- **规则引擎 `Game.gd`（1618 行）**：全部状态是普通变量，暴露 `action 方法 + can_* 守卫 + 派生查询`，只通过 **信号**（`changed/fx/game_over`）与视图通讯 → **完全可 headless 测试**。
- **视图 `GameScreen.gd`**：监听信号，每次刷新**全量程序化重建 UI**（每个 Button/Label 都是代码创建、硬编码坐标——便于确定性点击轨迹），**无 .tscn UI 布局**。
- **确定性**：所有 RNG 走 `_rng_at(round, salt)`，种子 = `seed_base + round*911 + salt` → 逐字节可复现回放。
- **数据驱动**：`data/*.json` id-keyed 字典（fish/regions/crew/tourists/tasks），效果由 `match-on-id` 代码解释；`balance.json` 单文件调参。
- **美术解析三级回退** `Art.pick`：`pro/<f>.png > <f>.png > <f>.svg` + `manifest.json`（40 槽位）→ 占位/正式美术零代码替换。

## 4. ⭐ 本地 LLM 集成（新项目要继承的核心资产）

这是 22nd 最宝贵、也最难、且**已实测跑通**的部分（doc 13 「AI 对手架构 · 本地 SLM 论证」+ `Game.gd`）。其结论与学界的 generative-agents 共识完全一致：

> **混合架构**：确定性引擎枚举**合法候选动作**、保证合法与强度；LLM 只在候选里**挑一个 + 用角色口吻说话**。
> 引擎永远兜底 → **再差/再任性的小模型也无法让游戏进入非法状态**。

**三档可插拔后端**（`Game.gd: AI_BACKENDS := ["logic","llm","slm"]`，title 处切换）：
| 后端 | 机制 | 用途 |
|---|---|---|
| `logic` | 确定性启发式 `ai_pick`（默认，唯一进 bench 的后端） | 强度地基、全平台、零依赖、bench 安全 |
| `llm` | OpenAI 兼容 chat（`HTTPRequest`，`user://ai_llm.json`，可填远程 key） | 接 LM Studio 或任意远程 API |
| `slm` | 本地 GGUF 经 **godot-llm（GDLlama）GDExtension**，后台 `Thread` 跑，`generate_text_json` + 语法约束 | 完全离线、开箱即用 |

接口契约：`ai_candidates() → ai_pick(intent) → ai_apply(intent)`；`_llm_pick()` 对 6 类脏输出兜底（抠 `{…}` 子串/越界回退/关键词/空串回退启发式）。

**实测硬结论（非推演，可直接套用到新项目）**：
1. **中文体积下限是 Qwen2.5-0.5B（~469–491MB Q4）**，不是 SmolLM2-360M——SmolLM2 是英文模型，中文直接崩。
2. **语法约束解码（json_schema / GBNF）是刚需**——否则 0.5B 会挑越界编号、把局面数字塞进台词。
3. **一句示例 +「别复述局面」指令** → 0.5B 实测 4/4 合法 + 4/4 像话，**无需微调**。
4. **延迟坑已解**：一次性 `generate_text_json` 每调用重载 469MB（~4.6s）；改 **instruct 常驻模式** → TURN1 538ms、TURN2 180ms（开局暖一次、之后 <0.6s/回合）。放**后台 Thread** + `call_deferred` 回主线程，不冻渲染。
5. **GDExtension 选 godot-llm（MIT，预编译 Win/Linux/Mac cpu+vulkan）** 或 **NobodyWho（EUPL-1.2，商用免费，自动 GBNF tool-calling、GPU）**；二者都预编译，无需逐平台自编译。
6. **Web 端不内置模型**（100–500MB 首载是 UX 灾难）→ 回退启发式 + 可选远程。
7. **bench 红线**：`scenario != ""` 的回放永远走确定性启发式；LLM/SLM 只在 `interactive`（`scenario==""`）下触发，绝不污染复现性。

> 这条「LLM 只当受约束的选择者 + 性格声音、引擎兜底合法」的纪律，是新项目 NPC 行为/对话的**直接蓝本**。

## 5. 对新项目的可复用清单（一句话版）

1. **常驻 Docker runner + exec/任务** 的快迭代评测环（Godot import 贵，保持温热）。
2. **HTML5/wasm 导出配方** + **`serve.js`** 零依赖托管（原样复用）。
3. **LLM-as-judge**：把多模态评审改造成「这个生活模拟看起来/表现对不对」的自动 reward 回路；`stub` judge 做免网 CI smoke。
4. **headless 自走子范式**：跑经济/需求/社会 soak 仿真，一次调一个旋钮。
5. **headless 规则引擎 + 信号驱动 UI** 的分离（sim 可无头快进）。
6. **确定性纪律**：salted seeded RNG + 把所有 LLM/交互/非确定特性 fence 在 `interactive` 之外。
7. **三档可插拔 LLM 后端** + `候选→挑选→应用` 契约 + 脏输出兜底（**最重要**）。
8. **数据驱动内容**（JSON id 字典 + 单 balance 文件）+ **美术三级回退**（占位像素图先行）。
9. 平衡的**自走子调参纪律**与 S/M/L **内容深度分档**、**轻量版 vs 1:1 忠实版双轨发布** 的工程节奏。

> 一个真正的差异点（见 doc 02/03）：《小鱼岛》是**回合制**、AI 只在「我方回合」算一次；新项目是**准实时多 Agent 生活模拟**，
> 必须叠加 generative-agents 的**记忆流 / 反思 / 规划 / tick 调度 / 导演-Agent 分层 / 成本控制**——这超出回合制卡牌 AI 的范畴，
> 是新项目设计的真正重心。
